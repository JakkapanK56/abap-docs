In the following, terms of the [ABAP Glossary](ABENABAP_GLOSSARY.html) are listed, that are relevant for the [ABAP Language](ABENABAP_REFERENCE.html).

**ABAP**

Advanced Business Application Programming. ABAP in the actual sense is the name of the [ABAP language](ABENABALA_GLOSRY.html), but in a broader sense it is also the term for the entire [ABAP technology](ABENABAP_TECHNO_GLOSRY.html) provided for and by an [ABAP system](ABENABAP_SYSTEM_GLOSRY.html).

**ABAP Cloud**

ABAP development model for state-of-the-art, cloud-ready and upgrade-stable solutions. Compared to [classic ABAP](ABENCLASSIC_ABAP_GLOSRY.html), ABAP Cloud uses the [ABAP technology](ABENABAP_TECHNO_GLOSRY.html) with the following restrictions:

The transactional programming model for ABAP Cloud is [RAP](ABENRAP_GLOSRY.html).

ABAP Cloud is supported in all SAP products that are based on ABAP technology. It is fully or partly mandatory in

On the [ABAP Platform](ABENABAP_PLATFORM_GLOSRY.html) of [SAP S/4HANA](ABENSAP_S4HANA_GLOSRY.html) it is available besides classic ABAP.

**ABAP command injection**

Injection of malicious ABAP statements by an attacker. An ABAP statement injection is possible if statements or parts of statements that are transferred from the outside and then are passed to the [runtime framework](ABENABAP_RUNTIME_FRMWK_GLOSRY.html) go unchecked.

**ABAP Compiler**

ABAP Compiler creates byte code as intermediate code when generating a program from [ABAP source code](ABENABAP_SOURCE_CODE_GLOSRY.html) that is stored in the database as a program load and is loaded into the [PXA](ABENPXA_GLOSRY.html) if required.

**ABAP connection**

Secondary or [service connection](ABENSERVICE_CONNECTION_GLOSRY.html) that is shared by [ABAP SQL](ABENABAP_SQL_GLOSRY.html), Native SQL and [AMDP](ABENABAP_MANAGED_DB_PROC_GLOSRY.html).

**binary stream**

[Stream](ABENSTREAM_GLOSRY.html) for [byte-like](ABENBYTE_LIKE_DATA_TYP_GLOSRY.html) data.

**bit expression**

Formulation of a binary operation in a [calculation expression](ABENCALCULATION_EXPRESSION_GLOSRY.html). The result of a bit expression is a byte-like value. Bit expressions occur in [read positions of specific statements](ABENOPERANDS_EXPRESSIONS.html).

**bit function**

[Built-in function](ABENBUILTIN_FUNCTION_GLOSRY.html) that processes individual bits of [byte chains](ABENBYTE_CHAIN_GLOSRY.html).

**bit operator**

Is applied to an [operand](ABENOPERAND_GLOSRY.html) or combines two operands of a [bit expression](ABENBIT_EXPRESSION_GLOSRY.html). The possible bit operators are [`BIT-NOT`](ABENBIT_OPERATORS.html), [`BIT-AND`](ABENBIT_OPERATORS.html), [`BIT-OR`](ABENBIT_OPERATORS.html), and [`BIT-XOR`](ABENBIT_OPERATORS.html).

**Boolean data object**

[Data object](ABENDATA_OBJECT_GLOSRY.html) used to register [truth values](ABENTRUTH_VALUE_GLOSRY.html). Boolean data objects are not supported in ABAP. Instead, a truth value can be evaluated as the return value of a [Boolean function](ABENBOOLE_FUNCTION_GLOSRY.html).

**Boolean function**

[Logical function](ABENLOGIC_FUNCTION_GLOSRY.html) that analyzes a [logical expression](ABENLOGICAL_EXPRESSION_GLOSRY.html) and returns the [truth value](ABENTRUTH_VALUE_GLOSRY.html) in character-like or byte-like form.

**Boolean operator**

Negates or combines [logical expressions](ABENLOGICAL_EXPRESSION_GLOSRY.html). Boolean operators belong to the [logical operators](ABENLOGICAL_OPERATOR_GLOSRY.html) that are used in [logical expressions](ABENLOGICAL_EXPRESSION_GLOSRY.html).

**bound data type**

[Data type](ABENDATA_TYPE_GLOSRY.html) that only exists as a property of a [data object](ABENDATA_OBJECT_GLOSRY.html). A bound data type has a [technical type name](ABENTECHNICAL_TYPE_NAME_GLOSRY.html) that can be retrieved as part of an [absolute type name](ABENABSOLUTE_TYPENAME_GLOSRY.html) by [RTTI](ABENRUN_TIME_TYPE_IDENTIFIC_GLOSRY.html). See also [standalone data type](ABENSTAND-ALONE_DATA_TYPE_GLOSRY.html).

**boxed component**

Structured [component](ABENCOMPONENT_GLOSRY.html) of a [structure](ABENSTRUCTURE_GLOSRY.html) or a [class](ABENCLASS_GLOSRY.html) or [interface](ABENOO_INTF_GLOSRY.html) that is not stored at the location of the definition but is [deep](ABENDEEP_GLOSRY.html), meaning it is managed by an internal reference. Boxed components are a mixture of [static](ABENSTATIC_DATA_OBJECT_GLOSRY.html) and [dynamic data objects](ABENDYNAMIC_DATA_OBJECT_GLOSRY.html). Currently, [static boxes](ABENSTATIC_BOX_GLOSRY.html) are supported as boxed components.

**branch**

Also referred to as selection. [Control structure](ABENCONTROL_STRUCTURE_GLOSRY.html) that can consist of multiple [statement blocks](ABENSTATEMENT_BLOCK_GLOSRY.html) that are defined using the keywords [`IF` - `ELSEIF` - `ELSE` - `ENDIF`](ABAPIF.html), [`CASE` - `WHEN` - `ENDCASE`](ABAPCASE.html), [`TRY` - `CATCH` - `CLEANUP` - `ENDTRY`](ABAPTRY.html) and are executed depending on conditions. Branches can be implemented in [operand positions](ABENOPERAND_POSITION_GLOSRY.html) using [conditional expressions](ABENCONDITIONAL_EXPRESSION_GLOSRY.html).

**built-in ABAP type**

[Built-in](ABENBUILTIN_DATA_TYPE_GLOSRY.html)\\ [elementary data type](ABENELEMENTARY_DATA_TYPE_GLOSRY.html). The built-in ABAP types are: [`b`](ABENBUILTIN_TYPES_NUMERIC.html), [`c`](ABENBUILTIN_TYPES_CHARACTER.html), [`d`](ABENBUILTIN_TYPES_DATE_TIME.html), [`decfloat16`](ABENBUILTIN_TYPES_NUMERIC.html), [`decfloat34`](ABENBUILTIN_TYPES_NUMERIC.html), [`f`](ABENBUILTIN_TYPES_NUMERIC.html), [`i`](ABENBUILTIN_TYPES_NUMERIC.html), [`int8`](ABENBUILTIN_TYPES_NUMERIC.html), [`n`](ABENBUILTIN_TYPES_CHARACTER.html), [`p`](ABENBUILTIN_TYPES_NUMERIC.html), [`s`](ABENBUILTIN_TYPES_NUMERIC.html), [`string`](ABENBUILTIN_TYPES_CHARACTER.html), [`t`](ABENBUILTIN_TYPES_DATE_TIME.html), [`utclong`](ABENBUILTIN_TYPES_DATE_TIME.html), [`x`](ABENBUILTIN_TYPES_BYTE.html), and [`xstring`](ABENBUILTIN_TYPES_BYTE.html).

[Data type](ABENDATA_TYPE_GLOSRY.html) that is provided in an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) by the [ABAP runtime framework](ABENABAP_RUNTIME_FRMWK_GLOSRY.html). A predefined function in ABAP, ABAP SQL, and ABAP CDS.

String or sequence of bytes. Content of any data object. However, direct access to the bytes of a byte chain is possible only in [byte-like data objects](ABENBYTE_LIKE_DATA_OBJECT_GLOSRY.html), or by using appropriately typed [field symbols](ABENFIELD_SYMBOL_GLOSRY.html).

[Built in function](ABENBUILTIN_FUNCTION_GLOSRY.html) for processing [byte chains](ABENBYTE_CHAIN_GLOSRY.html).

Result of the generation of an ABAP program using the ABAP Compiler. The statements in the byte code are linked with C functions. Determines the order in which the binary representation of a number of the [data types](ABENDATA_TYPE_GLOSRY.html)\\ [`i`](ABENBUILTIN_TYPES_NUMERIC.html), [`int8`](ABENBUILTIN_TYPES_NUMERIC.html), [`decfloat16`](ABENBUILTIN_TYPES_NUMERIC.html), [`decfloat34`](ABENBUILTIN_TYPES_NUMERIC.html), [`f`](ABENBUILTIN_TYPES_NUMERIC.html), [`s`](ABENBUILTIN_TYPES_NUMERIC.html), or a character is stored in the memory. A distinction is made between [big](ABENBIG_ENDIAN_GLOSRY.html) and [little endian](ABENLITTLE_ENDIAN_GLOSRY.html). In the former case, the system writes the byte with the highest level to the first position in the memory and in the latter case it writes the byte with the lowest level.

**field**

Another name for [elementary data object](ABENELEMENTARY_DATA_OBJECT_GLOSRY.html).

**field group**

Line structure of an extract dataset. Consists of program-global [fields](ABENFIELD_GLOSRY.html) of a program. Defined using the statement `FIELD-GROUPS`.

**field symbol**

Symbolic name for a [data object](ABENDATA_OBJECT_GLOSRY.html) to which actual memory areas can be assigned at program runtime. A field symbol can be used as a placeholder for a data object at an operand position. A field symbol is [typed](ABENTYPING_GLOSRY.html) either generically or completely. Field symbols are declared using the statement [`FIELD-SYMBOLS`](ABAPFIELD-SYMBOLS.html) or the [declaration operator](ABENDECLARATION_OPERATOR_GLOSRY.html)\\ [`FIELD-SYMBOL`](ABENFIELD-SYMBOL_INLINE.html); memory areas are assigned to them using [`ASSIGN`](ABAPASSIGN.html).

**file pointer**

Current position for writing to or reading from a file.

**filter operator**

[Constructor operator](ABENCONSTRUCTOR_OPERATOR_GLOSRY.html)\\ [`FILTER`](ABENCONSTRUCTOR_EXPRESSION_FILTER.html) for filtering lines of [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) using conditions.

**filter stream**

[Stream](ABENSTREAM_GLOSRY.html) that is not connected directly to a [streaming data source](ABENSTREAMING_DATA_SOURCE_GLOSRY.html) or a [streaming data sink](ABENSTREAMING_DATA_SINK_GLOSRY.html) but to another stream instead. A filter stream is either a [reader stream](ABENREADER_STREAM_GLOSRY.html) that reads and filters the data of another reader stream, or a [writer stream](ABENWRITER_STREAM_GLOSRY.html) that writes filtered data to another writer stream.

**final**

Term in [ABAP Objects](ABENABAP_OBJECTS_GLOSRY.html). A final [class](ABENCLASS_GLOSRY.html) cannot have any [subclasses](ABENSUBCLASS_GLOSRY.html). A final [method](ABENMETHOD_GLOSRY.html) cannot be [redefined](ABENREDEFINITION_GLOSRY.html).

**fixed point arithmetic**

Calculation with [packed numbers](ABENPACKED_NUMBER_GLOSRY.html) ([data type](ABENDATA_TYPE_GLOSRY.html)\\ [`p`](ABENBUILTIN_TYPES_NUMERIC.html)). Is calculated with an internal accuracy of 31 or (if not sufficient) 63 [places](ABENPLACE_GLOSRY.html). [Decimal places](ABENDECIMAL_PLACE_GLOSRY.html) are rounded arithmetically, if necessary. Places before the [decimal point](ABENDECIMAL_POINT_SEPARAT_GLOSRY.html) are never lost if a calculation is completed successfully. As a [program property](ABENPROGRAM_PROPERTY_GLOSRY.html), the *fixed point arithmetic* determines whether for numbers of type [`p`](ABENBUILTIN_TYPES_NUMERIC.html) the decimal point is respected in operations or not. The disabling of this program property is obsolete.

**fixed point number**

Number with a fixed number of [decimal places](ABENDECIMAL_PLACE_GLOSRY.html). In ABAP, fixed point numbers are implemented using [packed numbers](ABENPACKED_NUMBER_GLOSRY.html) where a sequence of digits contains a decimal separator at a position defined by a [packed number type](ABENPACKED_NUMBER_TYPE_GLOSRY.html).

**flat**

Property of a [data type](ABENDATA_TYPE_GLOSRY.html), where the content of its [data objects](ABENDATA_OBJECT_GLOSRY.html) represents the actual work data. All [elementary data types](ABENELEMENTARY_DATA_TYPE_GLOSRY.html) except [`string`](ABENBUILTIN_TYPES_CHARACTER.html) and [`xstring`](ABENBUILTIN_TYPES_BYTE.html) are flat. See also [deep](ABENDEEP_GLOSRY.html).

**flat structure**

[Structure](ABENSTRUCTURE_GLOSRY.html) that exclusively contains [flat](ABENFLAT_GLOSRY.html)\\ [structure components](ABENSTRUCTURE_COMPONENT_GLOSRY.html).

**flat table**

[Internal table](ABENINTERNAL_TABLE_GLOSRY.html) with a [flat](ABENFLAT_GLOSRY.html)\\ [line type](ABENROW_TYPE_GLOSRY.html).

**floating point arithmetic**

Calculation with [floating point numbers](ABENFLOATING_POINT_NUMBER_GLOSRY.html). It is executed directly on the [host computer](ABENHOST_COMPUTER_GLOSRY.html) of the current [AS instance](ABENAS_INSTANCE_GLOSRY.html). Either [binary floating point arithmetic](ABENBINFLOAT_ARITH_GLOSRY.html) or [decimal floating point arithmetic](ABENDECFLOAT_ARITH_GLOSRY.html).

**floating point function**

[Numeric function](ABENMATHEMATICAL_FUNKTION_GLOSRY.html) that expects a [floating point number](ABENFLOATING_POINT_NUMBER_GLOSRY.html) as an argument and returns a floating point number as a [return value](ABENRETURN_VALUE_GLOSRY.html).

**floating point number**

Either a [decimal floating point number](ABENDECFLOAT_GLOSRY.html) or a [binary floating point number](ABENBINFLOAT_GLOSRY.html). In floating point numbers, the number of [decimal places](ABENDECIMAL_PLACE_GLOSRY.html) is part of the value and not part of the data type, as is the case in [packed numbers](ABENPACKED_NUMBER_GLOSRY.html). [Floating point arithmetic](ABENFLOATING_POINT_ARITH_GLOSRY.html) is used for calculations with floating point numbers.

**friend**

Term in [ABAP Objects](ABENABAP_OBJECTS_GLOSRY.html). The additions [`FRIENDS`](ABAPCLASS_OPTIONS.html) and [`LOCAL FRIENDS`](ABAPCLASS_LOCAL_FRIENDS.html) of the statement [`CLASS`](ABAPCLASS_DEFINITION.html) define other [classes](ABENCLASS_GLOSRY.html) or [interfaces](ABENOO_INTF_GLOSRY.html) as friends of the class and thus grant them access to all of their components, regardless of the [visibility section](ABENVISIBILITY_SECTION_GLOSRY.html) or the addition [`READ-ONLY`](ABAPDATA_OPTIONS.html). In addition, friends of a class can create [instances](ABENINSTANCE_GLOSRY.html) of the class without restrictions.

**function group**

*\->*\\ [function pool](ABENFUNCTION_POOL_GLOSRY.html)

**function module**

[Procedure](ABENPROCEDURE_GLOSRY.html) that can only be implemented in [function pools](ABENFUNCTION_POOL_GLOSRY.html) and outside [classes](ABENCLASS_GLOSRY.html) between the statements [`FUNCTION`](ABAPFUNCTION.html) and [`ENDFUNCTION`](ABAPENDFUNCTION.html). An [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) can use the statement [`CALL FUNCTION`](ABAPCALL_FUNCTION.html) to call each function module allowed by the [package concept](ABENPACKAGE_CONCEPT_GLOSRY.html) externally.

**function module exit**

Customer exit for [ABAP programs](ABENABAP_PROGRAM_GLOSRY.html) in the form of empty or partially implemented [function modules](ABENFUNCTION_MODULE_GLOSRY.html) delivered with the standard SAP system. Integrated into ABAP programs using the statement `CALL CUSTOMER-FUNCTION`. The function module [interfaces](ABENINTF_GEN_GLOSRY.html) are predefined by SAP. The implementation can be empty or partially predefined. The customer must fully implement and activate them using transaction\\ `CMOD`. Function module exits should be replaced by [BAdIs](ABENBADI_GLOSRY.html).

**function pool**

The only kind of [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that can contain [function modules](ABENFUNCTION_MODULE_GLOSRY.html); usually loaded by calling one of its function modules. A function pool can currently contain a maximum of 99 function modules. It supports its own dynpros.

**functional method**

[Method](ABENMETHOD_GLOSRY.html) with exactly one [return value](ABENRETURN_VALUE_GLOSRY.html) declared using [`RETURNING`](ABAPMETHODS_FUNCTIONAL.html) and any number of other [formal parameters](ABENFORMAL_PARAMETER_GLOSRY.html). Can be used in [functional method calls](ABENFUNCTIONAL_METHOD_CALL_GLOSRY.html).

**functional method call**

An [expression](ABENEXPRESSION_GLOSRY.html) that calls a [functional method](ABENFUNCTIONAL_METHOD_GLOSRY.html) in a suitable [read position](ABENREAD_POSITION_GLOSRY.html), where the [return value](ABENRETURN_VALUE_GLOSRY.html) of the method is used as an [ABAP operand](ABENABAP_OPERAND_GLOSRY.html).

**functional operand position**

[Read position](ABENREAD_POSITION_GLOSRY.html) with any [operand type](ABENOPERAND_TYPE_GLOSRY.html), in which not only a suitable [data object](ABENDATA_OBJECT_GLOSRY.html) but also a [functional method call](ABENFUNCTIONAL_METHOD_CALL_GLOSRY.html) or [method chainings](ABENMETHOD_CHAINING_GLOSRY.html), a [constructor expression](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html), or a [table expression](ABENTABLE_EXPRESSION_GLOSRY.html) can be specified, the [return value](ABENRETURN_VALUE_GLOSRY.html) or result of which has the suitable [data type](ABENDATA_TYPE_GLOSRY.html).

**general expression position**

[Operand position](ABENREAD_POSITION_GLOSRY.html) in which not only a suitable [data object](ABENDATA_OBJECT_GLOSRY.html) can be specified, but also a [constructor expression](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html), a [table expression](ABENTABLE_EXPRESSION_GLOSRY.html), a [calculation expression](ABENCALCULATION_EXPRESSION_GLOSRY.html), a [built-in function](ABENBUILTIN_FUNCTION_GLOSRY.html), a [functional method](ABENFUNCTIONAL_METHOD_GLOSRY.html), or a [method chaining](ABENMETHOD_CHAINING_GLOSRY.html).

**general numeric function**

[Numeric function](ABENMATHEMATICAL_FUNKTION_GLOSRY.html) that expects a [numeric data object](ABENNUMERIC_DATA_OBJECT_GLOSRY.html) as an argument. The [data type](ABENDATA_TYPE_GLOSRY.html) of the argument determines the data type of the [return value](ABENRETURN_VALUE_GLOSRY.html) of the function.

**generic ABAP type**

[Built-in](ABENBUILTIN_DATA_TYPE_GLOSRY.html)\\ [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html). The generic ABAP types are: `any`, `any table`, `c`, `clike`, `csequence`, `data`, `decfloat`, `hashed table`, `index table`, `n`, `numeric`, `p`, `simple`, `sorted table`, `standard table`, `table`, `x`, and `xsequence`.

**generic typing**

[Typing](ABENTYPING_GLOSRY.html) in which the [data type](ABENDATA_TYPE_GLOSRY.html) of a [formal parameter](ABENFORMAL_PARAMETER_GLOSRY.html) or [field symbol](ABENFIELD_SYMBOL_GLOSRY.html) is not fully specified. The actual data type is copied at runtime from the assigned [actual parameter](ABENACTUAL_PARAMETER_GLOSRY.html) or memory area. During a dynamic access to a generically typed formal parameter or field symbol, the properties of the actual parameter apply. During static access, the properties of the actual parameter determined by the typing apply.

**geodata type**

Data type for geometric data. The [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html) currently supports the built-in data type [`GEOM_EWKB`](ABENDDIC_BUILTIN_TYPES.html).

**geospatial data**

*\->*\\ [spatial data](ABENSPATIAL_DATA_GLOSRY.html)

**global class**

[Class](ABENCLASS_GLOSRY.html) that is defined as a [global type](ABENGLOBAL_TYPE_GLOSRY.html). Global classes are declared in [class pools](ABENCLASS_POOL_GLOSRY.html) and are marked as such there using the addition [`PUBLIC`](ABAPCLASS_OPTIONS.html) of the statement [`CLASS`](ABAPCLASS.html). A class pool can only contain one global class.

**global declaration part**

Area following the opening statement of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) in which [interfaces](ABENOO_INTF_GLOSRY.html), [data types](ABENDATA_TYPE_GLOSRY.html), [classes](ABENCLASS_GLOSRY.html) and [data objects](ABENDATA_OBJECT_GLOSRY.html) that are visible in the whole program can be declared.

**global interface**

[Interface](ABENOO_INTF_GLOSRY.html) that is defined as a [global type](ABENGLOBAL_TYPE_GLOSRY.html). Global interfaces are declared in the [interface pools](ABENINTERFACE_POOL_GLOSRY.html) and are marked as such there using the addition `PUBLIC` of the statement [`INTERFACE`](ABAPINTERFACE.html). An interface pool cannot contain anything except for a global interface.

**global type**

[Data type](ABENDATA_TYPE_GLOSRY.html) or [object type](ABENOBJECT_TYPE_GLOSRY.html) that is visible as a [repository object](ABENREPOSITORY_OBJECT_GLOSRY.html) in the entire [AS ABAP](ABENAS_ABAP_GLOSRY.html). Global data types are defined as [DDIC types](ABENDDIC_TYPE_GLOSRY.html) in the [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html) or as [CDS types](ABENCDS_TYPE_GLOSRY.html) in [ABAP CDS](ABENABAP_CDS_GLOSRY.html). Global object types are defined as [global classes](ABENGLOBAL_CLASS_GLOSRY.html) or as [global interfaces](ABENGLOBAL_INTERFACE_GLOSRY.html) in the [class library](ABENCLASS_LIBRARY_GLOSRY.html). All these global types share the same namespace. Besides these global types, data types defined in the public [visibility section](ABENVISIBILITY_SECTION_GLOSRY.html) of global classes or in global interfaces can also be used globally.

**graph procedure**

A graph procedure is an [ABAP Managed Database Procedure](ABENABAP_MANAGED_DB_PROC_GLOSRY.html). It refers to [graph workspace](ABENGRAPH_WORKSPACE_GLOSRY.html) and operates on [graph](ABENGRAPH_GLOSRY.html) models. It is a read-only procedure that is written in [GraphScript](ABENGRAPHSCRIPT_GLOSRY.html).

**graph workspace**

A graph workspace is an object of the [SAP HANA database](ABENHANA_DATABASE_GLOSRY.html) that defines a [graph](ABENGRAPH_GLOSRY.html) in terms of tables and columns, i. e. it consists of two tables (for [vertices](ABENVERTEX_GLOSRY.html) and [edges](ABENEDGE_GLOSRY.html)) as sources for a graph model. It declares the key column of a [vertex table](ABENVERTEX_TABLE_GLOSRY.html) and an [edge table](ABENEDGE_TABLE_GLOSRY.html), and also the source and target column of an edge table. It is currently restricted to one key column and one source and target column. Graph workspaces are supported in ABAP by [AMDP](ABENAMDP_GLOSRY.html).

**group break**

Change from one [group level](ABENGROUP_LEVEL_GLOSRY.html) to another in [group level processing](ABENGROUP_LEVEL_PROCESSING_GLOSRY.html).

**group key**

Value of any [data type](ABENDATA_TYPE_GLOSRY.html) that is constructed for each read line in a group key expression when grouping [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) using [`LOOP AT itab ... GROUP BY`](ABAPLOOP_AT_ITAB_GROUP_BY.html) or [`FOR GROUPS ... OF`](ABENFOR_GROUPS_OF.html). All table lines for which the same value is produced are assigned to each group key. In [group level processing](ABENGROUP_LEVEL_PROCESSING_GLOSRY.html) for internal tables, the group key is the content of all columns to the left of a column specified after [`AT`](ABAPAT_ITAB.html).

**group key expression**

Expression for determining the group key of lines in the grouping of [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) with [`LOOP AT itab ... GROUP BY`](ABAPLOOP_AT_ITAB_GROUP_BY.html) or [`FOR GROUPS ... OF`](ABENFOR_GROUPS_OF.html).

**group level**

Set of lines in tabular data e.g. in an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) that have the same content in an initial part of the lines. Group levels are created by sorting the lines of the table by the initial part. The transition from one group level to the next is called a [group break](ABENGROUP_BREAK_GLOSRY.html). Group levels in internal tables can be evaluated with [group level processing](ABENGROUP_LEVEL_PROCESSING_GLOSRY.html). Group levels in [SQL data sources](ABENSQL_DATA_SOURCE_GLOSRY.html) can be transformed to [level-based hierarchies](ABENLEVEL_BASED_HIERA_GLOSRY.html) by the [hierarchy generator](ABENHIERARCHY_GENERATOR_GLOSRY.html) of [ABAP SQL](ABENABAP_SQL_GLOSRY.html).

**group level processing**

The evaluation of [group levels](ABENGROUP_LEVEL_GLOSRY.html) in [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) or the obsolete extracts. Group level processing takes place in [`LOOP`](ABAPLOOP_AT_ITAB.html)\-[loops](ABENLOOP_GLOSRY.html) using [control structures](ABENCONTROL_STRUCTURE_GLOSRY.html) defined using the keyword [`AT`](ABAPAT_ITAB.html).

**group loop**

Loop across the groups of an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) formed by the addition [`GROUP BY`](ABAPLOOP_AT_ITAB_GROUP_BY.html) of the statement [`LOOP AT`](ABAPLOOP_AT_ITAB_VARIANTS.html). A group loop can contain [member loops](ABENMEMBER_LOOP_GLOSRY.html) for the evaluation of the groups.

**grouping function**

[Aggregate function](ABENAGGREGATE_FUNCTION_GLOSRY.html) for [`GROUPING SETS`](ABAPGROUPING_SETS_CLAUSE.html) in a [query](ABENASQL_QUERY_GLOSRY.html). The result of the grouping function `GROUPING` indicates whether a specific column was part of the aggregation. The return value of the grouping function is 0 for the column that was not involved in the aggregation or 1 for the column that was involved in the aggregation.

**hash key**

Unique [table key](ABENTABLE_KEY_GLOSRY.html) of an [internal table](ABENINTERNAL_TABLE_GLOSRY.html), in which the assignment to the table lines is managed using a [hash algorithm](ABENHASH_ALGORITHM_GLOSRY.html). A hash key is the [primary table key](ABENPRIMARY_TABLE_KEY_GLOSRY.html) of a [hashed table](ABENHASHED_TABLE_GLOSRY.html) and can be assigned to every type of internal table as a [secondary table key](ABENPRIMARY_TABLE_KEY_GLOSRY.html). During access to an internal table using a hash key, the response time is constant, regardless of the number of table entries.

**hashed table**

[Table category](ABENTABLE_CATEGORY_GLOSRY.html) of an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) whose lines are stored internally using a [hash algorithm](ABENHASH_ALGORITHM_GLOSRY.html) and can be accessed using a unique [hash key](ABENHASH_KEY_GLOSRY.html). A hashed table does not have a [primary table index](ABENPRIMARY_TABLE_INDEX_GLOSRY.html), but it can be assigned a [secondary table index](ABENSECONDARY_TABLE_INDEX_GLOSRY.html) using a [secondary table key](ABENSECONDARY_TABLE_KEY_GLOSRY.html). The corresponding [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`hashed table`](ABENBUILT_IN_TYPES_GENERIC.html).

**header line**

Obsolete [work area](ABENWORK_AREA_GLOSRY.html) of an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) whose type is the [line type](ABENROW_TYPE_GLOSRY.html) and has the same name as the internal table. When an internal table with a header line is used in an [operand position](ABENOPERAND_POSITION_GLOSRY.html), the header line is usually addressed. To force access to the [table body](ABENTABLE_BODY_GLOSRY.html), square brackets `[]` can be specified after the table name.

**heap**

Also referred to as free or dynamic memory. Part of the [program memory](ABENROLL_AREA_GLOSRY.html), where [anonymous data objects](ABENANONYMOUS_DATA_OBJECT_GLOSRY.html) created dynamically using [`CREATE DATA`](ABAPCREATE_DATA.html), [`CREATE OBJECT`](ABAPCREATE_OBJECT.html), or [`NEW`](ABENCONSTRUCTOR_EXPRESSION_NEW.html) or [instances](ABENINSTANCE_GLOSRY.html) of [classes](ABENCLASS_GLOSRY.html) are stored. The occupied memory can be released again by the [garbage collector](ABENGARBAGE_COLLECTOR_GLOSRY.html). See also [stack](ABENSTACK_GLOSRY.html).

**heap reference**

[Reference](ABENREFERENCE_GLOSRY.html) to an [object](ABENOBJECT_GLOSRY.html) or part of an object in the [heap](ABENHEAP_GLOSRY.html). A heap reference keeps an object alive. See also [stack reference](ABENSTACK_REFERENCE_GLOSRY.html).

**host constant**

[Host variable](ABENHOST_VARIABLE_GLOSRY.html) for which the ABAP [data object](ABENDATA_OBJECT_GLOSRY.html) is a [constant](ABENCONSTANT_GLOSRY.html).

**host expression**

ABAP-[expression](ABENEXPRESSION_GLOSRY.html) specified in an operand position of an [ABAP SQL](ABENABAP_SQL_HOST_EXPRESSIONS.html) statement. Host expressions are always specified in parentheses preceded by the [escape character](ABENESCAPE_CHARACTER_GLOSRY.html)\\ `@`: `@( ... )`.

**host variable**

ABAP [data object](ABENDATA_OBJECT_GLOSRY.html) specified in an operand position of an [ABAP SQL](ABENABAP_SQL_HOST_VARIABLES.html) statement or of a statically embedded Native SQL statement. The ABAP data object can be a [variable](ABENVARIABLE_GLOSRY.html) or a [constant](ABENCONSTANT_GLOSRY.html) (see [host constant](ABENHOST_CONSTANT_GLOSRY.html)). In ABAP SQL, a host variable must be prefixed with the [escape character](ABENESCAPE_CHARACTER_GLOSRY.html)\\ `@` ; in Native SQL with the escape character `:`. In ADBC, the placeholder `?` to which ABAP data objects are bound, is used instead of host variables.

**immutable variable**

[Variable](ABENVARIABLE_GLOSRY.html) that is declared with the [declaration operator](ABENDECLARATION_OPERATOR_GLOSRY.html)\\ [`FINAL`](ABENFINAL_INLINE.html). This [inline declaration](ABENINLINE_DECLARATION_GLOSRY.html) is the only [write position](ABENWRITE_POSITION_GLOSRY.html) within a context where a value can be assigned to an immutable variable.

**implementation part**

Part of the definition of a [class](ABENCLASS_GLOSRY.html) in which the [methods](ABENMETHOD_GLOSRY.html) of the class are implemented.

**importing parameter**

*\->*\\ [input parameter](ABENINPUT_PARAMETER_GLOSRY.html)

**include program**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that cannot be compiled independently but must be included in another ABAP program using the statement `INCLUDE`. To be able to be compiled, an include program must be part of a [compilation unit](ABENCOMPILATION_UNIT_GLOSRY.html). An include program should only be assigned to a single compilation unit.

**index table**

General term for [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) that are managed using a [primary table index](ABENPRIMARY_TABLE_INDEX_GLOSRY.html) ([standard tables](ABENSTANDARD_TABLE_GLOSRY.html) and [sorted tables](ABENSORTED_TABLE_GLOSRY.html)). A [secondary table key](ABENSECONDARY_TABLE_KEY_GLOSRY.html) can be assigned to enable access using a [hash algorithm](ABENHASH_ALGORITHM_GLOSRY.html). The corresponding [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`index table`](ABENBUILT_IN_TYPES_GENERIC.html).

**indicator structure**

[Substructure](ABENSUBSTRUCTURE_GLOSRY.html) of a [structured type](ABENSTRUCTURED_TYPE_GLOSRY.html). The indicator structure is the last component of the structured type and mirrors the preceding components. The components of the substructure serve as indicators for features of the preceding components. Besides actual indicator structures, there are also [condensed indicator structures](ABENCONDENSED_IND_STRUCTURE_GLOSRY.html) that are realized by [byte fields](ABENBYTE_FIELD_GLOSRY.html) where the individual bits serve as indicators. Indicator structures can be defined with the addition [`WITH INDICATORS`](ABAPTYPES_INDICATORS.html) of the statement [`TYPES`](ABAPTYPES_INDICATORS.html) and can be used as [ABAP SQL indicators](ABENABAP_SQL_INDICATOR_GLOSRY.html). In the latter case, also [inline declarations](ABENINLINE_DECLARATION_GLOSRY.html) are possible.

**inheritance**

Generalization-specialization relationship between [classes](ABENCLASS_GLOSRY.html). Inheritance means that new [subclasses](ABENSUBCLASS_GLOSRY.html) are derived by adopting all [class components](ABENCLASS_COMPONENT_GLOSRY.html) from [superclasses](ABENSUPERCLASS_GLOSRY.html). A subclass can also be made more specific by declaring new components and [redefining](ABENREDEFINITION_GLOSRY.html)\\ [methods](ABENMETHOD_GLOSRY.html).

**inheritance hierarchy**

Hierarchical relationship between [superclasses](ABENSUPERCLASS_GLOSRY.html) and [subclasses](ABENSUBCLASS_GLOSRY.html) in [inheritance](ABENINHERITANCE_GLOSRY.html) functions. When [single inheritance](ABENSINGLE_INHERITANCE_GLOSRY.html) is used, the inheritance hierarchy forms an [inheritance tree](ABENINHERITANCE_TREE_GLOSRY.html).

**inheritance tree**

[Inheritance hierarchy](ABENINHERITANCE_HIERARCHY_GLOSRY.html) of the [single inheritance](ABENSINGLE_INHERITANCE_GLOSRY.html) in which a unique path leads from each [subclass](ABENSUBCLASS_GLOSRY.html) to a [root class](ABENROOT_CLASS_GLOSRY.html).

**initial memory requirement**

Property of a [table type](ABENTABLE_TYPE_GLOSRY.html) that affects the memory area allocated by the ABAP runtime framework for an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) before the table is filled with lines. The initial memory requirement is not a [technical type property](ABENTECHNICAL_TYPE_PRPT_GLOSRY.html) and does not affect the [compatibility](ABENCOMPATIBLE_GLOSRY.html) of the table type.

**initial value**

The ABAP runtime framework defines an initial value for each [data type](ABENDATA_TYPE_GLOSRY.html). The initial values of the [elementary data types](ABENELEMENTARY_DATA_TYPE_GLOSRY.html) depend on the corresponding [built-in ABAP type](ABENBUILT_IN_TYPES_COMPLETE.html). The initial value of [reference types](ABENREFERENCE_TYPE_GLOSRY.html) is the [null reference](ABENNULL_REFERENCE_GLOSRY.html). The initial value of an [internal table](ABENINTERNAL_TABLE_GLOSRY.html) is an empty table without lines. The initial value of [structured types](ABENSTRUCTURED_TYPE_GLOSRY.html) is produced from the initial values of the individual [structure components](ABENSTRUCTURE_COMPONENT_GLOSRY.html). The statement [`CLEAR`](ABAPCLEAR.html) with no additions sets a [data object](ABENDATA_OBJECT_GLOSRY.html) to the appropriate initial value for its type. Initial values are also assigned to most [built-in data types](ABENDDIC_BUILTIN_TYPES.html) in [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html), but those are not evaluated in ABAP programs.

**initial value sharing**

The status of a [boxed component](ABENBOXED_COMPONENT_GLOSRY.html) as long as it is processed by read access only. The [initial value](ABENINITIAL_VALUE_GLOSRY.html) of all type-compliant boxed components of the same type is only stored once in each [AS instance](ABENAS_INSTANCE_GLOSRY.html) in the [PXA](ABENPXA_GLOSRY.html).

**injection**

Test-friendly code located between the statements [`TEST-INJECTION`](ABAPTEST-INJECTION.html) and [`END-TEST-INJECTION`](ABAPEND-TEST-INJECTION.html) in a [method](ABENMETHOD_GLOSRY.html) of a [test class](ABENTEST_CLASS_GLOSRY.html). Injections replace the associated [test seams](ABENTEST_SEAM_GLOSRY.html) when a [unit test](ABENUNIT_TEST_GLOSRY.html) is executed.

**inline declaration**

[Declaration](ABENDECLARATION_GLOSRY.html) of a [variable](ABENVARIABLE_GLOSRY.html), an [immutable variable](ABENIMMUTABLE_VARIABLE_GLOSRY.html) or a [field symbol](ABENFIELD_SYMBOL_GLOSRY.html) using a [declaration expression](ABENDECLARATION_EXPRESSION_GLOSRY.html) in a [declaration position](ABENDECLARATION_POSITION_GLOSRY.html).

**input parameter**

[Formal parameter](ABENFORMAL_PARAMETER_GLOSRY.html) to which the value of an [actual parameter](ABENACTUAL_PARAMETER_GLOSRY.html) is passed at the time of a call.

**input stream**

*\->*\\ [Reader stream](ABENREADER_STREAM_GLOSRY.html)

**input/output parameter**

[Formal parameter](ABENFORMAL_PARAMETER_GLOSRY.html) to which the value of an [actual parameter](ABENACTUAL_PARAMETER_GLOSRY.html) is passed at the time of the call. It can be changed during the call and is available in the actual parameter at the end of the call.

**insertion function**

[Processing function](ABENPROCESS_FUNCTION_GLOSRY.html) that inserts a string into another string.

**instance**

Concrete occurrence of an abstract [type](ABENTYPE_GLOSRY.html). The instance of a [data type](ABENDATA_TYPE_GLOSRY.html) is a [data object](ABENDATA_OBJECT_GLOSRY.html). The instance of a [class](ABENCLASS_GLOSRY.html) is often simply described as an object. See also [instantiation](ABENINSTANTIATION_GLOSRY.html).

**instance attribute**

[Attribute](ABENATTRIBUTE_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) that is declared using [`DATA`](ABAPDATA.html). Instance attributes are only valid in the [context](ABENOBJ_CONTEXT_GLOSRY.html) of an [instance](ABENINSTANCE_GLOSRY.html) of the class. The content of the instance attributes determines the state of the [objects](ABENOBJECT_GLOSRY.html) of a class.

**instance component**

Superordinate term for [instance attributes](ABENINSTANCE_ATTRIBUTE_GLOSRY.html), [instance events](ABENINSTANCE_EVENT_GLOSRY.html), and [instance methods](ABENINSTANCE_METHOD_GLOSRY.html). The instance components of a [class](ABENCLASS_GLOSRY.html) can only be addressed in [instances](ABENINSTANCE_GLOSRY.html) of the class.

**instance component selector**

*\->*\\ [object component selector](ABENOBJECT_COMPONENT_SELECT_GLOSRY.html)

**instance constructor**

[Constructor](ABENCONSTRUCTOR_GLOSRY.html) declared under the name [`constructor`](ABENCONSTRUCTOR.html) as an [instance method](ABENINSTANCE_METHOD_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html), and which is automatically called when the class is [instantiated](ABENINSTANTIATION_GLOSRY.html) and after the [object](ABENOBJECT_GLOSRY.html) is created.

**instance event**

[Event](ABENEVENT_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) declared using [`EVENTS`](ABAPEVENTS.html). Instance events usually indicate status changes of [objects](ABENOBJECT_GLOSRY.html).

**instance method**

[Method](ABENMETHOD_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) declared using [`METHODS`](ABAPMETHODS.html), which can only be used in an [instance](ABENINSTANCE_GLOSRY.html) of the class. Instance methods can access all [attributes](ABENATTRIBUTE_GLOSRY.html) and [events](ABENEVENT_GLOSRY.html) of their class.

**instance operator**

[Constructor operator](ABENCONSTRUCTOR_OPERATOR_GLOSRY.html)\\ [`NEW`](ABENCONSTRUCTOR_EXPRESSION_NEW.html) used to [instantiate](ABENINSTANTIATION_GLOSRY.html)\\ [objects](ABENOBJECT_GLOSRY.html).

**instantiation**

Creation of an [instance](ABENINSTANCE_GLOSRY.html). [Data objects](ABENDATA_OBJECT_GLOSRY.html) as instances of [data types](ABENDATA_TYPE_GLOSRY.html) are created implicitly when a [context](ABENOBJ_CONTEXT_GLOSRY.html) is loaded, or explicitly using [`CREATE DATA`](ABAPCREATE_DATA.html) or [`NEW`](ABENCONSTRUCTOR_EXPRESSION_NEW.html). [Objects](ABENOBJECT_GLOSRY.html) as an instance of [classes](ABENCLASS_GLOSRY.html) are only created explicitly using [`CREATE OBJECT`](ABAPCREATE_OBJECT.html) or [`NEW`](ABENCONSTRUCTOR_EXPRESSION_NEW.html).

**integer**

[Numeric data object](ABENNUMERIC_DATA_OBJECT_GLOSRY.html) of ABAP or elementary object of the database with an [integer type](ABENINTEGER_TYPE_GLOSRY.html).

**integer literal**

A [numeric literal](ABENNUMERIC_LITERAL_GLOSRY.html) of an [integer type](ABENINTEGER_TYPE_GLOSRY.html).

See also:

**integer type**

[Numeric data type](ABENNUMERIC_DATA_TYPE_GLOSRY.html) for storing [integer numbers](ABENINTEGER_NUMBER_GLOSRY.html) of different value ranges. The associated [built-in ABAP types](ABENBUILTIN_ABAP_TYPE_GLOSRY.html) are ([`b`](ABENBUILTIN_TYPES_NUMERIC.html), [`s`](ABENBUILTIN_TYPES_NUMERIC.html)), [`i`](ABENBUILTIN_TYPES_NUMERIC.html), and [`int8`](ABENBUILTIN_TYPES_NUMERIC.html). The [built-in DDIC types](ABENBUILTIN_DDIC_TYPE_GLOSRY.html) are [`INT1`](ABENDDIC_BUILTIN_TYPES.html#abenddic-builtin-types-2-----int2---abenddic-builtin-types-html-@ITOC@@ABENDDIC_BUILTIN_TYPES_2), [`INT4`](ABENDDIC_BUILTIN_TYPES.html#abenddic-builtin-types-2--and---int8---abenddic-builtin-types-html-@ITOC@@ABENDDIC_BUILTIN_TYPES_2). The internal representation takes place in the integer representation of the current platform. A [data object](ABENDATA_OBJECT_GLOSRY.html) of an integer type is an [integer](ABENINTEGER_GLOSRY.html).

**interface (ABAP Objects)**

Template for the [public](ABENPUBLIC_GLOSRY.html)\\ [interface](ABENINTF_GEN_GLOSRY.html) of [classes](ABENCLASS_GLOSRY.html) in [ABAP Objects](ABENABAP_OBJECTS_GLOSRY.html). Defined with [`INTERFACE - ENDINTERFACE`](ABAPINTERFACE.html), either as a [global interface](ABENGLOBAL_INTERFACE_GLOSRY.html) in an [interface pool](ABENINTERFACE_POOL_GLOSRY.html) or as a [local interface](ABENLOCAL_INTERFACE_GLOSRY.html) in another [ABAP program](ABENABAP_PROGRAM_GLOSRY.html). An interface contains the declaration of [interface components](ABENINTERFACE_COMPONENT_GLOSRY.html) but does not contain method implementations. Can be implemented in classes using [`INTERFACES`](ABAPINTERFACES.html), which enhances the external interface of the class by adding interface components. OO interfaces are generally referred to as interfaces, unless the distinction to [general interfaces](ABENINTF_GEN_GLOSRY.html) needs to be made.

**interface (general)**

Boundary between two [contexts](ABENOBJ_CONTEXT_GLOSRY.html).

**interface component**

[Component](ABENCOMPONENT_GLOSRY.html) of an [interface](ABENOO_INTF_GLOSRY.html). Declared in the [declaration part](ABENDECLARATION_PART_GLOSRY.html) of an interface. Possible interface components are: [Attributes](ABENATTRIBUTE_GLOSRY.html), [methods](ABENMETHOD_GLOSRY.html), and [events](ABENEVENT_GLOSRY.html).

**interface component selector**

Character `~`. An [interface component](ABENINTERFACE_COMPONENT_GLOSRY.html)\\ `comp` of an [interface](ABENOO_INTF_GLOSRY.html)\\ `intf` can be addressed in an implementing [class](ABENCLASS_GLOSRY.html) or a [compound interface](ABENCOMPOUND_INTERFACE_GLOSRY.html) via `intf~comp`.

**interface pool**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that contains the definition of exactly one [global interface](ABENGLOBAL_INTERFACE_GLOSRY.html) and is loaded when using the interface. Does not support its own dynpros.

**interface reference variable**

[Object reference variable](ABENOBJECT_REFER_VARIABLE_GLOSRY.html) of the [static type](ABENSTATIC_TYPE_GLOSRY.html) of an [interface](ABENOO_INTF_GLOSRY.html). An interface reference variable can be used to access the visible [interface components](ABENINTERFACE_COMPONENT_GLOSRY.html) of an object that exists in the corresponding interface. This applies to both dynamic and static access.

**interface work area**

Special [data objects](ABENDATA_OBJECT_GLOSRY.html) that can be used as a cross-program [interface](ABENINTF_GEN_GLOSRY.html) between programs and dynpros, and between programs and logical databases. Interface work areas are either table work areas or common areas (obsolete). All programs in a [program group](ABENPROGRAM_GROUP_GLOSRY.html) access the same data of an interface work area.

**internal mode**

Obsolete name for [internal session](ABENINTERNAL_SESSION_GLOSRY.html).

**internal procedure call**

Call of a [processing block](ABENPROCESSING_BLOCK_GLOSRY.html) ([procedure](ABENPROCEDURE_GLOSRY.html)) of the same program. The required program does not have to be reloaded into the [internal session](ABENINTERNAL_SESSION_GLOSRY.html).

**internal session**

Instance within an [ABAP session](ABENABAP_SESSION_GLOSRY.html) with its own [program memory](ABENROLL_AREA_GLOSRY.html) in which the data and [objects](ABENOBJECT_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) are stored during its execution. An internal session is created when an ABAP program is called in the [shared memory](ABENSHARED_MEMORY_GLOSRY.html) and is preserved for as long as the main program of its main program group is executed.

**internal table**

[Dynamic data object](ABENDYNAMIC_DATA_OBJECT_GLOSRY.html) that consists of a sequence of lines of the same [data type](ABENDATA_TYPE_GLOSRY.html). An internal table has a [table type](ABENTABLE_TYPE_GLOSRY.html) that specifies the [line type](ABENROW_TYPE_GLOSRY.html), the [table category](ABENTABLE_CATEGORY_GLOSRY.html), and a [table key](ABENTABLE_KEY_GLOSRY.html). The corresponding [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`any table`](ABENBUILT_IN_TYPES_GENERIC.html).

**iteration**

Repeated processing of a [statement block](ABENSTATEMENT_BLOCK_GLOSRY.html) in a [loop](ABENLOOP_GLOSRY.html) or an [expression](ABENEXPRESSION_GLOSRY.html) with an [iteration expression](ABENITERATION_EXPRESSION_GLOSRY.html).

**iteration expression**

Expression that executes an [iteration](ABENITERATION_GLOSRY.html). An iteration expression is a subexpression of a [constructor expression](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html) introduced using the [iteration operator](ABENITERATION_OPERATOR_GLOSRY.html)\\ [`FOR`](ABENFOR.html) and executes [conditional iterations](ABENFOR_CONDITIONAL.html) or [table iterations](ABENTABLE_ITERATION_GLOSRY.html).

**iteration operator**

The iteration operator `FOR` introduces an [iteration expression](ABENITERATION_EXPRESSION_GLOSRY.html) as a subexpression of a [constructor expression](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html).

**kernel method**

[Method](ABENMETHOD_GLOSRY.html) that is implemented in the [kernel](ABENKERNEL_GLOSRY.html) of the [ABAP runtime framework](ABENABAP_RUNTIME_FRMWK_GLOSRY.html) instead of in an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html).

**key table**

Superordinate term for [sorted tables](ABENSORTED_TABLE_GLOSRY.html) and [hashed tables](ABENHASHED_TABLE_GLOSRY.html), in which the [primary table key](ABENPRIMARY_TABLE_KEY_GLOSRY.html), unlike [standard tables](ABENSTANDARD_TABLE_GLOSRY.html), has its own stable administration that allows optimized access.

**language environment**

Language-dependent settings for [ABAP programs](ABENABAP_PROGRAM_GLOSRY.html). The language environment includes both [language settings](ABENLANGUAGE_SETTING_GLOSRY.html) and [format settings](ABENFORMAT_SETTING_GLOSRY.html).

**language setting**

Part of the [language environment](ABENLANGUAGE_ENVIRONMENT_GLOSRY.html). The language setting of an ABAP program comprises the [logon language](ABENLOGON_LANGUAGE_GLOSRY.html) and the [text environment language](ABENTEXT_ENV_LANGU_GLOSRY.html) of the [text environment](ABENTEXT_ENVIRONMENT_GLOSRY.html).

**lazy update**

Procedure to create or update non-unique [secondary table keys](ABENSECONDARY_TABLE_KEY_GLOSRY.html) in [internal tables](ABENINTERNAL_TABLE_GLOSRY.html). Non-unique secondary tables keys are not updated straight after insert operations using [`INSERT`](ABAPINSERT_ITAB.html) or [`APPEND`](ABAPAPPEND.html), a block operation in which the whole table body is filled at once, or after a modifying operation using [`MODIFY`](ABAPMODIFY_ITAB.html), [field symbols](ABENFIELD_SYMBOL_GLOSRY.html), or [data references](ABENDATA_REFERENCE_GLOSRY.html). They are updated at the next explicit usage. See also [direct update](ABENDIRECT_UPDATE_GLOSRY.html) and [delayed update](ABENDELAYED_UPDATE_GLOSRY.html).

**legacy file**

File on a [host computer](ABENHOST_COMPUTER_GLOSRY.html) of an [AS instance](ABENAS_INSTANCE_GLOSRY.html) that was opened using the addition `LEGACY` of the statement `OPEN DATASET`. In a legacy file, the [byte order](ABENBYTE_ORDER_GLOSRY.html) and the [code page](ABENCODEPAGE_GLOSRY.html) for handling the content of the file can be specified.

**length function**

[Description function](ABENDESCRIPTION_FUNCTION_GLOSRY.html) that determines the length of a data object in characters or bytes.

**`LET` expression**

A `LET` expression is a subexpression of a [constructor expression](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html) that declares [variables](ABENVARIABLE_GLOSRY.html) or [field symbols](ABENFIELD_SYMBOL_GLOSRY.html) as local helper fields.

**line type**

[Technical type property](ABENTECHNICAL_TYPE_PRPT_GLOSRY.html) of a [table type](ABENTABLE_TYPE_GLOSRY.html) that defines the common [data type](ABENDATA_TYPE_GLOSRY.html) of the lines in an [internal table](ABENINTERNAL_TABLE_GLOSRY.html).

**literal**

The textual representation of a value in the [ABAP source code](ABENABAP_SOURCE_CODE_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) or in a [CDS object](ABENCDS_OBJECT_GLOSRY.html). There are [typed literals](ABENTYPED_LITERAL_GLOSRY.html) and [untyped literals](ABENUNTYPED_LITERAL_GLOSRY.html) available.

See also:

**literal operator**

The literal operator `&` joins two [text field literals](ABENTEXT_FIELD_LITERAL_GLOSRY.html) or two [text string literals](ABENTEXT_STRING_LITERAL_GLOSRY.html) as a single literal or two [string templates](ABENSTRING_TEMPLATE_GLOSRY.html) as a single string template. The literal operator is supported in [ABAP literals](ABENABAP_LITERAL_GLOSRY.html) and in [ABAP SQL literals](ABENABAP_SQL_LITERAL_GLOSRY.html), but not in [CDS literals](ABENCDS_LITERAL_GLOSRY.html).

**load**

In ABAP, a load is a binary representation of a [repository object](ABENREPOSITORY_OBJECT_GLOSRY.html), which is optimized for fast access, in the memory or on the database. Examples of loads are program load that is made up of the byte code of the existing program or the package load.

**local class**

[Class](ABENCLASS_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that can be accessed statically from the ABAP program only. Local classes can be created in all program types, with the exception of [interface pools](ABENINTERFACE_POOL_GLOSRY.html) and [type pools](ABENTYPE_POOL_GLOSRY.html). In particular, [class pools](ABENCLASS_POOL_GLOSRY.html) can contain local classes for use in the [global class](ABENGLOBAL_CLASS_GLOSRY.html) or by frameworks. All local classes of an ABAP program share the same namespace together with the [local interfaces](ABENLOCAL_INTERFACE_GLOSRY.html) and the program-global [data types](ABENDATA_TYPE_GLOSRY.html) declared with [`TYPES`](ABAPTYPES.html).

**local data**

Local data is declared in [procedures](ABENPROCEDURE_GLOSRY.html) and is only known there. See also global data.

**local data queue**

Persistence layer into which data can be placed and read only in the order in which it was stored. The local data queue triggers the no-send scenario of the qRFC.

**local interface**

[Interface](ABENOO_INTF_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that can be accessed statically in this program only. Local interfaces can be created in all program types except for [interface pools](ABENINTERFACE_POOL_GLOSRY.html) and [type pools](ABENTYPE_POOL_GLOSRY.html). All local interfaces of an ABAP program share the same namespace together with the [local classes](ABENLOCAL_CLASS_GLOSRY.html) and the program-global [data types](ABENDATA_TYPE_GLOSRY.html) declared using [`TYPES`](ABAPTYPES.html).

**locator**

Pointer to substrings of [LOBs](ABENLOB_GLOSRY.html) in a [database](ABENDATABASE_GLOSRY.html). In ABAP, locators are [LOB handles](ABENLOB_HANDLE_GLOSRY.html) and are implemented using [instances](ABENINSTANCE_GLOSRY.html) of the corresponding [system classes](ABENSYSTEM_CLASS_GLOSRY.html). Locators can be linked with the database in question in [ABAP SQL statements with LOBs](ABENABAP_SQL_GLOSRY.html). Different [locator types](ABENLOCATOR_TYPE_GLOSRY.html) are provided for [BLOBs](ABENBLOB_GLOSRY.html) and [CLOBs](ABENCLOB_GLOSRY.html).

**logical database**

Special [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that provides other ABAP programs with data from the nodes of a hierarchical tree structure (obsolete). Is either linked to an executable program in its [program properties](ABENPROGRAM_PROPERTY_GLOSRY.html) or is called using [function module](ABENFUNCTION_MODULE_GLOSRY.html)\\ `LDB_PROCESS`. A logical database is made up of a hierarchical structure with nodes, an ABAP database program and its own standard selection screen. Logical databases are maintained in the Logical Database Builder.

**logical expression**

Formulation of a condition for [operands](ABENOPERAND_GLOSRY.html). A logical expression is a single [relational expression](ABENRELATIONAL_EXPRESSION_GLOSRY.html) or is made up of multiple relational expressions combined by [Boolean operators](ABENBOOLEAN_OPERATOR_GLOSRY.html). The result of a logical expression is a [truth value](ABENTRUTH_VALUE_GLOSRY.html), that is, either true or false.

**logpoint**

In an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html), a logpoint is defined statically as an unconditional [checkpoint](ABENCHECKPOINT_GLOSRY.html) using the statement `LOG-POINT`, or dynamically in transaction\\ `SDLP` or in the [ABAP development tools for Eclipse (ADT)](ABENADT_GLOSRY.html). When an active logpoint is reached, a log entry is created. A static logpoint must be assigned to a [checkpoint group](ABENCHECKPOINT_GROUP_GLOSRY.html) when it is created. This group controls its [activation](ABENACTIVATABLE_CHECKPOINT_GLOSRY.html).

**loop**

[Control structure](ABENCONTROL_STRUCTURE_GLOSRY.html) that performs an [iteration](ABENITERATION_GLOSRY.html). Consists of a [statement block](ABENSTATEMENT_BLOCK_GLOSRY.html) defined by the keywords [`DO` - `ENDDO`](ABAPDO.html), [`WHILE` - `ENDWHILE`](ABAPWHILE.html), [`LOOP` - `ENDLOOP`](ABAPLOOP_AT_ITAB.html), `PROVIDE` - `ENDPROVIDE`, [`SELECT` - `ENDSELECT`](ABAPSELECT.html) and that can be executed multiple times.

**lossless assignment**

[Assignment](ABENASSIGNMENT_GLOSRY.html) between two non-[compatible](ABENCOMPATIBLE_GLOSRY.html)\\ [data objects](ABENDATA_OBJECT_GLOSRY.html), where the source field contains a valid value, the [conversion](ABENCONVERSION_GLOSRY.html) is lossless, and the target field contains a valid value after the assignment. The [lossless operator](ABENLOSSLESS_OPERATOR_GLOSRY.html)\\ [`EXACT`](ABENCONSTRUCTOR_EXPRESSION_EXACT.html) and the addition `EXACT` of the obsolete statement `MOVE`\\ [](ABENASSIGNMENT_GLOSRY.html)can be used to check whether an assignment is lossless.

**lossless calculation**

[Arithmetic](ABENARITHMETIC_EXPRESSION_GLOSRY.html) calculation where no rounding occurs. The [lossless operator](ABENLOSSLESS_OPERATOR_GLOSRY.html)\\ [`EXACT`](ABENCONSTRUCTOR_EXPRESSION_EXACT.html) or the addition `EXACT` of the obsolete statement `COMPUTE` can be used to check whether a calculation is lossless.

**lossless operator**

[Constructor operator](ABENCONSTRUCTOR_OPERATOR_GLOSRY.html)\\ [`EXACT`](ABENCONSTRUCTOR_EXPRESSION_EXACT.html) for lossless creation of a value for a [data type](ABENDATA_TYPE_GLOSRY.html).

**macro**

Combination of a statement sequence for internal reuse of [source code](ABENSOURCE_CODE_GLOSRY.html) within a program. In [ABAP source code](ABENABAP_SOURCE_CODE_GLOSRY.html), macros are defined between `DEFINE` and `END-OF-DEFINITION` or in database table `TRMAC`.

**main program**

The first program of a [program group](ABENPROGRAM_GROUP_GLOSRY.html). The first program (executable program, module pool, or [function pool](ABENFUNCTION_POOL_GLOSRY.html)) that is loaded by a program call into an [internal session](ABENINTERNAL_SESSION_GLOSRY.html) is the main program of the main program group. The program (function pool or [class pool](ABENCLASS_POOL_GLOSRY.html)), for which an additional program group is created when it is loaded is the main program of the additional program group.

**main program group**

First [program group](ABENPROGRAM_GROUP_GLOSRY.html) of an [internal session](ABENINTERNAL_SESSION_GLOSRY.html). It is created when the internal session is created through a program call. The first program in an internal session is the main program of the main program group. The runtime of the main program in the main program group determines its lifespan and that of the entire internal session.

**main session**

Obsolete name for [ABAP session](ABENABAP_SESSION_GLOSRY.html)

**master program**

[Compilation unit](ABENCOMPILATION_UNIT_GLOSRY.html) that is modularized into different [processing blocks](ABENPROCESSING_BLOCK_GLOSRY.html) which are distributed over several [include programs](ABENINCLUDE_PROGRAM_GLOSRY.html). Typical master programs are [class](ABENCLASS_POOL_GLOSRY.html) and [function pools](ABENFUNCTION_POOL_GLOSRY.html), that are created implicitly by the development tools. The [ABAP Unit](ABENABAP_UNIT_GLOSRY.html) tests for the units in the include programs are usually created in the [test include](ABENTEST_INCLUDE_GLOSRY.html) of the master program.

**match function**

[Processing function](ABENPROCESS_FUNCTION_GLOSRY.html) that returns a substring of a character string that matches a [regular expression](ABENREGULAR_EXPRESSION_GLOSRY.html).

**measuring interval**

Duration of execution of a limited measuring section.

**measuring section**

A sequence of statements delimited by `GET RUN TIME` in the program code, whose runtime is to be measured.

**member loop**

Loop at the members of a group in a [group loop](ABENGROUP_LOOP_GLOSRY.html) introduced using the statement [`LOOP AT GROUP`](ABAPLOOP_AT_GROUP.html).

**mesh**

Instance of a [mesh type](ABENMESH_TYPE_GLOSRY.html). Special [structure](ABENSTRUCTURE_GLOSRY.html) whose [components](ABENSTRUCTURE_COMPONENT_GLOSRY.html) described as [mesh nodes](ABENMESH_NODE_GLOSRY.html) are exclusively [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) that can be joined using [mesh associations](ABENMESH_ASSOCIATION_GLOSRY.html).

**mesh association**

[Association](ABENASSOCIATION_GLOSRY.html) of a [mesh](ABENMESH_GLOSRY.html). Property of a [mesh node](ABENMESH_NODE_GLOSRY.html) and therefore part of the [mesh type](ABENMESH_TYPE_GLOSRY.html). Associates mesh nodes in [mesh paths](ABENMESH_PATH_GLOSRY.html).

**mesh node**

Component of a [mesh](ABENMESH_GLOSRY.html). Mesh nodes are always [internal tables](ABENINTERNAL_TABLE_GLOSRY.html) with a structured [line type](ABENROW_TYPE_GLOSRY.html). Mesh nodes can have [mesh associations](ABENMESH_ASSOCIATION_GLOSRY.html) that they can address in [mesh paths](ABENMESH_PATH_GLOSRY.html).

**mesh path**

Chaining of multiple [mesh nodes](ABENMESH_NODE_GLOSRY.html) using their [mesh associations](ABENMESH_ASSOCIATION_GLOSRY.html). A mesh path describes a set of lines in its final path node. These lines are accessed when a mesh path is used.

**mesh path expression**

A standalone specification of a [mesh path](ABENMESH_PATH_GLOSRY.html) in an [operand position](ABENOPERAND_POSITION_GLOSRY.html) for which the same rules apply as for a [table expression](ABENTABLE_EXPRESSION_GLOSRY.html).

**mesh type**

[Data type](ABENDATA_TYPE_GLOSRY.html) of a [mesh](ABENMESH_GLOSRY.html). Special [structured type](ABENSTRUCTURED_TYPE_GLOSRY.html) that contains only [table types](ABENTABLE_TYPE_GLOSRY.html) with structured [line types](ABENROW_TYPE_GLOSRY.html) as [components](ABENSTRUCTURE_COMPONENT_GLOSRY.html) that can be linked using [mesh associations](ABENMESH_ASSOCIATION_GLOSRY.html).

**message**

Text that can be accessed using the statement [`MESSAGE`](ABAPMESSAGE.html). Messages are always displayed in the [logon language](ABENLOGON_LANGUAGE_GLOSRY.html), which cannot be changed during a [user session](ABENUSER_SESSION_GLOSRY.html). Message texts are organized into [message classes](ABENMESSAGE_CLASS_GLOSRY.html) and [message numbers](ABENMESSAGE_NUMBER_GLOSRY.html) and are maintained in the transaction\\ message maintenance. When a message is accessed using the statement [`MESSAGE`](ABAPMESSAGE.html), it is classified with a [message type](ABENMESSAGE_TYPE_GLOSRY.html) that defines the behavior of the message. Messages are also used as exception texts of [exception classes](ABENEXCEPTION_CLASS_GLOSRY.html).

**message class**

Identifier that groups [messages](ABENMESSAGE_GLOSRY.html) in an application area.

**message number**

Number that identifies the [messages](ABENMESSAGE_GLOSRY.html) of a [message class](ABENMESSAGE_CLASS_GLOSRY.html).

**message type**

Must be specified when the statement [`MESSAGE`](ABAPMESSAGE.html) is used to send a [message](ABENMESSAGE_GLOSRY.html). It determines the representation of the message in the SAP GUI and the subsequent program flow. The possible message types are [information message](ABENINFORMATION_MESSAGE_GLOSRY.html), [status message](ABENSTATUS_MESSAGE_GLOSRY.html), [error message](ABENERROR_MESSAGE_GLOSRY.html), [warning](ABENWARNING_GLOSRY.html), [termination message](ABENTERMINATION_MESSAGE_GLOSRY.html), and [exit message](ABENEXIT_MESSAGE_GLOSRY.html).

**method**

[Procedure](ABENPROCEDURE_GLOSRY.html) that is declared as a [component](ABENCOMPONENT_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) or of an [interface](ABENOO_INTF_GLOSRY.html). The two possible methods are [instance methods](ABENINSTANCE_METHOD_GLOSRY.html) and [static methods](ABENSTATIC_METHOD_GLOSRY.html). A method can be implemented only in the [implementation part](ABENIMPLEMENTATION_PART_GLOSRY.html) of its class, between [`METHOD`](ABAPMETHOD.html) and [`ENDMETHOD`](ABAPENDMETHOD.html).

**method chaining**

Chained calls of [functional methods](ABENFUNCTIONAL_METHOD_GLOSRY.html). Either a [chained method call](ABENCHAINED_METHOD_CALL_GLOSRY.html) or a [chained attribute access](ABENCHAINED_ATTRIBUTE_ACC_GLOSRY.html).

**module pool**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html) that normally contains dynpros and dialog modules, and is started using transaction codes.

**multiplication assignment operator**

[Calculation assignment operator](ABENCALC_ASSIGNMENT_OP_GLOSRY.html)\\ `*=` that multiplies values in an assignment.

**multitenancy**

Principle of software architecture in which a single software instance is run for different tenants on a single server. In a multitenancy [AS ABAP](ABENAS_ABAP_GLOSRY.html), [application clients](ABENAPPLICATION_CLIENT_GLOSRY.html) assume the role of tenants. Multitenancy is not supported by the [ABAP runtime framework](ABENABAP_RUNTIME_FRMWK_GLOSRY.html) in the current release.

**named data object**

[Data object](ABENDATA_OBJECT_GLOSRY.html) that can be identified via a name. Named data objects are [constants](ABENCONSTANT_GLOSRY.html), [variables](ABENVARIABLE_GLOSRY.html), and [text symbols](ABENTEXT_SYMBOL_GLOSRY.html) and can be stored in the [stack](ABENSTACK_GLOSRY.html) or as [instance attributes](ABENINSTANCE_ATTRIBUTE_GLOSRY.html) in the [heap](ABENHEAP_GLOSRY.html). See also [unnamed data object](ABENUNNAMED_DATA_OBJECT_GLOSRY.html).

**named include**

*\->*\\ [component group](ABENCOMPONENT_GROUP_GLOSRY.html)

**narrowing cast**

*\->*\\ [Downcast](ABENDOWN_CAST_GLOSRY.html)

**nested interface**

*\->*\\ [Compound interface](ABENCOMPOUND_INTERFACE_GLOSRY.html)

**nested structure**

[Structure](ABENSTRUCTURE_GLOSRY.html) with one or more [substructures](ABENSUBSTRUCTURE_GLOSRY.html). A nested structure can contain [boxed components](ABENBOXED_COMPONENT_GLOSRY.html).

**non-blocking mode**

Any statements that block the program flow are not allowed in an [ABAP programs](ABENABAP_PROGRAM_GLOSRY.html) that is processed in non-blocking mode. There are values of the non-blocking mode for APC and ADF.

**Non-Unicode ABAP**

Obsolete [language version](ABENABAP_VERSION_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) in which the [Unicode checks](ABENUNICODE_CHECK_GLOSRY.html) are not effective. A program with this language version cannot be executed in [Unicode systems](ABENUNICODE_SYSTEM_GLOSRY.html). In the current [AS ABAP](ABENAS_ABAP_GLOSRY.html) release, only Unicode systems are supported. Therefore, Non-Unicode ABAP can no longer be used.

**non-Unicode program**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html) written in [ABAP language version](ABENABAP_VERSION_GLOSRY.html)\\ Non-Unicode ABAP.

**null indicator**

[ABAP SQL indicator](ABENABAP_SQL_INDICATOR_GLOSRY.html) in a structured target area of an [ABAP SQL](ABENABAP_SQL_GLOSRY.html)\\ [query](ABENASQL_QUERY_GLOSRY.html). The components or bits of a null indicator indicate which columns of the result set contain the [null value](ABENNULL_VALUE_GLOSRY.html) and which do not.

**null reference**

[Initial value](ABENINITIAL_VALUE_GLOSRY.html) of a [reference variable](ABENREFERENCE_VARIABLE_GLOSRY.html). The null reference does not point to an [object](ABENOBJECT_GLOSRY.html) and can be assigned to any reference variable. [Data objects](ABENDATA_OBJECT_GLOSRY.html) accessed using the null reference generally produce an [uncatchable exception](ABENUNCATCHABLE_EXCEPTION_GLOSRY.html). Method calls using the null reference produce a [catchable exception](ABENCATCHABLE_EXCEPTION_GLOSRY.html).

**numeric data object**

[Data object](ABENDATA_OBJECT_GLOSRY.html) that can be interpreted numerically if the content is appropriate. In addition to data objects with [numeric data types](ABENNUMERIC_DATA_TYPE_GLOSRY.html), [byte-like](ABENBYTE_LIKE_DATA_OBJECT_GLOSRY.html) and [character-like data objects](ABENCHARLIKE_DATA_OBJECT_GLOSRY.html) can also be numeric.

**numeric data type**

[Data type](ABENDATA_TYPE_GLOSRY.html) whose [data objects](ABENDATA_OBJECT_GLOSRY.html) contain numeric values in platform-dependent encoding. These data types are [integer type](ABENINTEGER_TYPE_GLOSRY.html), [packed number type](ABENPACKED_NUMBER_TYPE_GLOSRY.html), [decimal floating pointing number type](ABENDECFLOAT_TYPE_GLOSRY.html) and [binary floating pointing number type](ABENBINFLOAT_TYPE_GLOSRY.html). The corresponding [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`numeric`](ABENBUILT_IN_TYPES_GENERIC.html).

**numeric expression**

Represents a numeric value. A numeric expression is a [constructor expression](ABENCONSTRUCTOR_GLOSRY.html), [arithmetic expression](ABENARITHMETIC_EXPRESSION_GLOSRY.html), [table expression](ABENTABLE_EXPRESSION_GLOSRY.html), or a [function](ABENFUNCTION_GLOSRY.html) whose return value has a [numeric data type](ABENNUMERIC_DATA_TYPE_GLOSRY.html).

**numeric expression position**

[Read position](ABENREAD_POSITION_GLOSRY.html) in which not only a [numeric data object](ABENNUMERIC_DATA_OBJECT_GLOSRY.html) that can be converted into the operand type can be specified, but also a [numeric expression](ABENNUMERICAL_EXPRESSION_GLOSRY.html).

**numeric function**

[Built-in function](ABENBUILTIN_FUNCTION_GLOSRY.html) that calculates numeric values.

**numeric literal**

A [literal](ABENABAP_LITERAL_GLOSRY.html) that has a [numeric data type](ABENNUMERIC_DATA_TYPE_GLOSRY.html) and whose characters represent a numeric value.

See also:

**numeric operand position**

[Operand position](ABENOPERAND_POSITION_GLOSRY.html) with numeric [operand type](ABENOPERAND_TYPE_GLOSRY.html), in which a [numeric data object](ABENNUMERIC_DATA_OBJECT_GLOSRY.html) that can be converted into the operand type can be specified. See also [numeric expression position](ABENNUMERICAL_EXPR_POSITION_GLOSRY.html).

**numeric text field**

[Character-like data object](ABENCHARLIKE_DATA_OBJECT_GLOSRY.html) of ABAP or elementary object of the database with a [numeric text field type](ABENNUMERIC_TEXT_FIELD_TYPE_GLOSRY.html).

**numeric text field type**

[Character-like data type](ABENCHARLIKE_DATA_TYPE_GLOSRY.html) for storing [numeric texts](ABENNUMERIC_TEXT_GLOSRY.html) of fixed length. The associated [built-in ABAP type](ABENBUILTIN_ABAP_TYPE_GLOSRY.html) is [`n`](ABENBUILTIN_TYPES_CHARACTER.html). The [built-in DDIC types](ABENBUILTIN_DDIC_TYPE_GLOSRY.html) are [`NUMC`](ABENDDIC_BUILTIN_TYPES.html#abenddic-builtin-types-7--and---accp---abenddic-builtin-types-html-@ITOC@@ABENDDIC_BUILTIN_TYPES_6). The internal representation takes place in the [character representation](ABENCHAR_REPRESENTATION_GLOSRY.html) of the current [code page](ABENCODEPAGE_GLOSRY.html). A [data object](ABENDATA_OBJECT_GLOSRY.html) of this type is a [numeric text field](ABENNUMERIC_TEXT_FIELD_GLOSRY.html).

**object**

[Instance](ABENINSTANCE_GLOSRY.html) of a [type](ABENTYPE_GLOSRY.html). Either an instance of a [class](ABENCLASS_GLOSRY.html) or, as a [data object](ABENDATA_OBJECT_GLOSRY.html), an instance of a [data type](ABENDATA_TYPE_GLOSRY.html). Exists in the [internal session](ABENINTERNAL_SESSION_GLOSRY.html) of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) or as a shared object in the [shared memory](ABENSHARED_MEMORY_GLOSRY.html). There are also persistent objects in the [database](ABENDATABASE_GLOSRY.html).

**object component selector**

Character `->`. Also referred to as [instance component selector](ABENINSTANCE_COMP_SELECTOR_GLOSRY.html). Each visible [component](ABENCOMPONENT_GLOSRY.html)\\ `comp` of a [data object](ABENDATA_OBJECT_GLOSRY.html) or an [instance](ABENINSTANCE_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) can be addressed using `ref->comp` if `ref` is a [reference variable](ABENREFERENCE_VARIABLE_GLOSRY.html) that refers to the object.

**object context**

Environment of an [object](ABENOBJECT_GLOSRY.html). Determines its visibility and lifetime. Possible contexts are [procedures](ABENPROCEDURE_GLOSRY.html), [instances](ABENINSTANCE_GLOSRY.html) of [classes](ABENCLASS_GLOSRY.html), classes, and [programs](ABENABAP_PROGRAM_GLOSRY.html).

**object plug-in**

Object plug-ins are used to make modification-free functional [enhancements](ABENENHANCEMENT_GLOSRY.html) to ABAP programs. They are instances of [BAdI implementation classes](ABENBADI_IMPLEMENT_CLASS_GLOSRY.html) and are administered by [BAdI objects](ABENBADI_OBJECT_GLOSRY.html). Object plug-ins are created when the [`GET BADI`](ABAPGET_BADI.html) statement is executed and in accordance with the filter specifications.

**object reference**

[Reference](ABENREFERENCE_GLOSRY.html) that points to an [instance](ABENINSTANCE_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html). An non-initial object reference is always a [heap reference](ABENHEAP_REFERENCE_GLOSRY.html). See also [weak reference](ABENWEAK_REFERENCE_GLOSRY.html).

**object reference variable**

[Reference variable](ABENREFERENCE_VARIABLE_GLOSRY.html) for [object references](ABENOBJECT_REFERENCE_GLOSRY.html). Object reference variables can be either [class reference variables](ABENCLASS_REFERENCE_VARIABL_GLOSRY.html) or [interface reference variables](ABENINTERFACE_REF_VARIABLE_GLOSRY.html).

**Object Services**

OS for short. [Interfaces](ABENOO_INTF_GLOSRY.html) and [classes](ABENCLASS_GLOSRY.html) from the [class library](ABENCLASS_LIBRARY_GLOSRY.html) with the prefixes `IF_OS_` and `CL_OS_`. Object services provide persistence services and transaction services that are used to manage persistent objects and object-oriented transactions. There is also a query service for finding and loading persistent objects.

**object transaction**

The transaction code of an object transaction is linked with a [method](ABENMETHOD_GLOSRY.html) of a local or global [class](ABENCLASS_GLOSRY.html). When the transaction is called, the associated program is loaded, for [instance methods](ABENINSTANCE_METHOD_GLOSRY.html) an [object](ABENOBJECT_GLOSRY.html) of the class is created, and the method is executed. Object transactions can be connected to a transaction service of the Object Services.

**object type**

Description of an [object](ABENOBJECT_GLOSRY.html) in [ABAP Objects](ABENABAP_OBJECTS_GLOSRY.html). Either a [class](ABENCLASS_GLOSRY.html) for the entire object or [interface](ABENOO_INTF_GLOSRY.html) for a part of an object. The associated [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`object`](ABENBUILT_IN_TYPES_GENERIC.html). Object types are either defined as [program global type](ABENPROGRAM_GLOBAL_TYPE_GLOSRY.html) in an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) or as a [global type](ABENGLOBAL_TYPE_GLOSRY.html) in the [class library](ABENCLASS_LIBRARY_GLOSRY.html).

**obsolete language element**

[ABAP language element](ABENABAP_LANGUAGE_ELEMENT_GLOSRY.html) that still exists in the language for reasons of downward compatibility, but should no longer be used. Most obsolete language elements are allowed only outside [classes](ABENCLASS_GLOSRY.html).

**offset/length specification**

Access to subareas of a [data object](ABENDATA_OBJECT_GLOSRY.html) by appending `[+off][(len)]` to the name of the data object.

**operand**

A component of an [ABAP statement](ABENABAP_STATEMENT_GLOSRY.html) or of a [CDS data definition](ABENCDS_DATA_DEFINITION_GLOSRY.html) at an [operand position](ABENOPERAND_POSITION_GLOSRY.html). An operand can be an [elementary operand](ABENELEMENTARY_OPERAND_GLOSRY.html) or an [expression](ABENEXP_GLOSRY.html).

There are different types of operands:

**operand position**

Position of an [ABAP statement](ABENABAP_STATEMENT_GLOSRY.html) or of a [CDS data definition](ABENCDS_DATA_DEFINITION_GLOSRY.html) at which an [operand](ABENOPERAND_GLOSRY.html) is listed. There are [read](ABENREAD_POSITION_GLOSRY.html) and [write positions](ABENWRITE_POSITION_GLOSRY.html). In an ABAP statement, such an operand position is typed with an [operand type](ABENOPERAND_TYPE_GLOSRY.html) and appropriate [data objects](ABENDATA_OBJECT_GLOSRY.html) or [expressions](ABENEXPRESSION_GLOSRY.html) can be specified. The operand positions for the latter are [expression positions](ABENEXPRESSION_POSITION_GLOSRY.html). Besides that there are other operand positions, where types, callable units or other external resources such as [CDS entities](ABENCDS_ENTITY_GLOSRY.html) or [DDIC database tables](ABENDDIC_DB_TABLE_GLOSRY.html) can be specified as operands.

**operand type**

[Data type](ABENDATA_TYPE_GLOSRY.html) with which an [operand position](ABENOPERAND_POSITION_GLOSRY.html) is typed. The operand type can be complete or generic. An [operand](ABENOPERAND_GLOSRY.html) must match the operand type. Depending on the statement, [type conversions](ABENTYPE_CONVERSION_GLOSRY.html) or [castings](ABENCAST_CASTING_GLOSRY.html) take place for incompatible operands. In [inline declarations](ABENINLINE_DECLARATION_GLOSRY.html), the operand types determine the declared data type. For [constructor expressions](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html), the operand type can be used to infer the type of the result.

**operator**

Component of an [ABAP statement](ABENABAP_STATEMENT_GLOSRY.html) or of a [CDS data definition](ABENCDS_DATA_DEFINITION_GLOSRY.html) that, when combined with [operands](ABENOPERAND_GLOSRY.html), forms an [expression](ABENEXP_GLOSRY.html). When the statement is executed, an operation is executed for two operands connected by one operator, and the result is processed further in the statement. There are [arithmetic](ABENARITHMETIC_OPERATOR_GLOSRY.html), [bit](ABENBIT_OPERATOR_GLOSRY.html), [string](ABENSTRING_OPERATOR_GLOSRY.html), [Boolean](ABENBOOLEAN_OPERATOR_GLOSRY.html), [relational](ABENREL_OPERATOR_GLOSRY.html), and [assignment operators](ABENASSIGNMENT_OPERATOR_GLOSRY.html). There are also operators that do not combine operands, but perform other operations in [operand positions](ABENOPERAND_POSITION_GLOSRY.html). Operators of this type are the [dereferencing operator](ABENDEREFERENCING_OPERAT_GLOSRY.html), [declarations operators](ABENDECLARATION_OPERATOR_GLOSRY.html), and [constructor operators](ABENCONSTRUCTOR_OPERATOR_GLOSRY.html). Finally, the [literal operator](ABENASSIGNMENT_OPERATOR_GLOSRY.html) is used to combine two [literals](ABENABAP_LITERAL_GLOSRY.html).

**output parameter**

[Formal parameter](ABENFORMAL_PARAMETER_GLOSRY.html) whose value is set during the call and is made available in the [actual parameter](ABENACTUAL_PARAMETER_GLOSRY.html) after the call.

**output statement**

Statement for writing into a list. Output statements are `WRITE` and `ULINE`. There are also statements for positioning the list cursor.

**output stream**

*\->*\\ [Writer stream](ABENWRITER_STREAM_GLOSRY.html)

**packed number literal**

A [numeric literal](ABENNUMERIC_LITERAL_GLOSRY.html) of [packed number type](ABENPACKED_NUMBER_TYPE_GLOSRY.html).

**transaction**

Execution of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html) using a transaction code. There are dialog, report, parameter, variant, and `OO` transactions. A transaction is started by entering the transaction code in the command field on the standard toolbar or by using the statements `CALL TRANSACTION` or `LEAVE TO TRANSACTION`.

At SAP, database transactions are referred to as [database LUWs](ABENDATABASE_LUW_GLOSRY.html).

In a business context, for example in [RAP](ABENRAP_GLOSRY.html), a transaction is refers to the result of a business-related operation that accesses and modifies data. When completed, a consistent state of data is produced.

**transaction service**

**transactional contract**

Specifies which ABAP statements and operations are allowed and which are not allowed in a [transactional phase](ABENTR_PHASE_GLOSRY.html). The following transactional contracts are available, each of which specifies restrictions to ensure transactional consistency: `SAVE`, `MODIFY`, `FUNCTIONAL`, `SURFACE`, `READ`, `COMPLETE`. The `SAVE` and `MODIFY` contracts are implicitly enabled in both transactional phases.

The transactional contracts can be set explicitly by [API classifications](ABAPAPI_CLASSIFICATION.html) (starting with `IF_ABAP_TX...`, for example, `IF_ABAP_TX_SAVE`). In the [controlled SAP LUW](ABENCONTROLLED_SAP_LUW_GLOSRY.html) and when in a transactional context, transactional contracts are checked.

For example, an API contains database changes and has the classification `IF_ABAP_TX_SAVE`. The classification ensures that the API is not called in the modify [transactional phase](ABENTR_PHASE_GLOSRY.html) by causing a violation that is either logged or results in a [runtime error](ABENRUNTIME_ERROR_GLOSRY.html).

**transactional phase**

Specifies the concrete phase of a transaction. It is either the modify or the save transactional phase. They are set either implicitly, as in the case of, for example, [RAP handler](ABENABP_HANDLER_METHOD_GLOSRY.html) and [saver](ABENABP_SAVER_METHOD_GLOSRY.html) methods, or explicitly using the static methods of class [`CL_ABAP_TX`](ABAPRAP_CL_ABAP_TX.html).

In a [RAP transaction](ABENRAP_LUW_GLOSRY.html), the transactional phases are as follows:

**transactional Remote Function Call**

tRFC for short. Obsolete remote function call in which the function called remotely is registered for execution and started using the statement [`COMMIT WORK`](ABAPCOMMIT.html).

**transformation source code**

[Source code](ABENSOURCE_CODE_GLOSRY.html) of [XSLT programs](ABENXSLT_PROGRAM_GLOSRY.html) and [Simple Transformations](ABENSIMPLE_TRANSFORMATION_GLOSRY.html) processed in the Transformation Editor.

**translate function**

[Processing function](ABENPROCESS_FUNCTION_GLOSRY.html) that translates characters in a string into other characters.

**tRFC**

*\->*\\ transactional remote function call

**trusted system**

[AS ABAP](ABENAS_ABAP_GLOSRY.html) that is classified as a trusted caller of remote function calls in a trusting system. The trusted system provides the RFC client. The system is classified in transaction\\ `SMT1`. Trusted systems log on to the trusting system without passing a password. User-specific logon data is checked in the trusting system.

**trusting system**

[AS ABAP](ABENAS_ABAP_GLOSRY.html) that trusts a trusted system in calls using Remote Function Call. The trusting system provides the RFC server. The trusting systems can be listed in a trusted system using transaction code\\ `SMT2`.

**truth value**

A truth value is either true or false. In ABAP, truth values only occur as the result of [logical expressions](ABENLOGICAL_EXPRESSION_GLOSRY.html). [Data types](ABENDATA_TYPE_GLOSRY.html) for [Boolean data objects](ABENBOOLEAN_DATA_OBJECT_GLOSRY.html) that can accept truth values are not supported. Instead, a truth value can be evaluated as the result of a [Boolean function](ABENBOOLE_FUNCTION_GLOSRY.html).

**type**

Abstract description of an [object](ABENOBJECT_GLOSRY.html). Either a [data type](ABENDATA_TYPE_GLOSRY.html) for a [data object](ABENDATA_OBJECT_GLOSRY.html) or an [object type](ABENOBJECT_TYPE_GLOSRY.html) for an object in [ABAP Objects](ABENABAP_OBJECTS_GLOSRY.html). The associated [generic data type](ABENGENERIC_DATA_TYPE_GLOSRY.html) is [`any`](ABENBUILT_IN_TYPES_GENERIC.html). All types share the same [namespace](ABENNAMING_CONVENTIONS.html).

**type conversion**

Converts the content of a [data object](ABENDATA_OBJECT_GLOSRY.html) according to the associated [conversion rules](ABENCONVERSION_RULE_GLOSRY.html). Type conversions occur implicitly in [assignments](ABENASSIGNMENT_GLOSRY.html) between data objects with different [data types](ABENDATA_TYPE_GLOSRY.html) or can be performed explicitly using the [conversion operator](ABENCONVERSION_OPERATOR_GLOSRY.html)\\ [`CONV`](ABENCONSTRUCTOR_EXPRESSION_VALUE.html).

**type description class**

[System class](ABENSYSTEM_CLASS_GLOSRY.html) of the [runtime type services](ABENRUN_TIME_TYPE_SERVICES_GLOSRY.html). The [static methods](ABENSTATIC_METHOD_GLOSRY.html) of a type description classes create [type description objects](ABENTYPE_OBJECT_GLOSRY.html) from existing types or construct new types.

**type description object**

Instance of a [type description class](ABENTYPE_CLASS_GLOSRY.html). Type description objects describe a [type](ABENTYPE_GLOSRY.html). Type description objects are created using the static methods of the type description classes. The type description class and possible attributes of type description objects determine the [technical properties](ABENTECHNICAL_TYPE_PRPT_GLOSRY.html) of the type. [References](ABENREFERENCE_GLOSRY.html) to type description objects can be used, for example, after the addition `HANDLE` of the statements [`CREATE DATA`](ABAPCREATE_DATA_HANDLE.html) and [`ASSIGN`](ABAPASSIGN_CASTING.html).

**type group**

*\->*\\ [type pool](ABENTYPE_POOL_GLOSRY.html)

**type name**

Name of a [type](ABENTYPE_GLOSRY.html). In a program, only the [relative type name](ABENRELATIVE_TYPE_NAME_GLOSRY.html) is known and statically usable. An [absolute type name](ABENABSOLUTE_TYPENAME_GLOSRY.html) can be used to specify dynamic types.

**type pool**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html), administrated by the [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html), that is introduced by the statement `TYPE-POOL` and which contains the definitions of globally visible [data types](ABENDATA_TYPE_GLOSRY.html), [constants](ABENCONSTANT_GLOSRY.html), and macros. A type pool is loaded the first time its data types and constants are used. It does not support own dynpros.

**typing**

[Generic](ABENGENERIC_TYPING_GLOSRY.html) or [complete](ABENCOMPLETE_TYPING_GLOSRY.html) definition of a [data type](ABENDATA_TYPE_GLOSRY.html) for [formal parameters](ABENFORMAL_PARAMETER_GLOSRY.html) or [field symbols](ABENFIELD_SYMBOL_GLOSRY.html). Specifies in which operand positions a formal parameter or field symbol can be used and which type properties are used. When parameters are passed, it is checked whether the data type of the assigned [actual parameter](ABENACTUAL_PARAMETER_GLOSRY.html), or memory area is [compatible](ABENCOMPATIBLE_GLOSRY.html) with the typing.

**uncatchable exception**

[Exception](ABENEXCEPTION_GLOSRY.html) that is not assigned to an [exception class](ABENEXCEPTION_CLASS_GLOSRY.html) and that cannot be handled in the program by [`TRY - CATCH - ENDTRY`](ABAPTRY.html). Uncatchable exceptions are raised by the [ABAP runtime framework](ABENABAP_RUNTIME_FRMWK_GLOSRY.html) and are usually a response to error situations that cannot be handled properly in the program. An uncatchable exception always produces a [runtime error](ABENRUNTIME_ERROR_GLOSRY.html).

**Unicode**

Language-independent [character set](ABENCHARACTER_SET_GLOSRY.html) for international data processing ([www.unicode.org](http://www.unicode.org)). Subset of the [`UCS`](ABENUCS_GLOSRY.html) character set according to ISO-10646. Unicode can represent 1114111 different characters numbered hexadecimally from `U+0000` to `U+10FFFF`. The Unicode character set is normally accessed using a [UTF](ABENUTF_GLOSRY.html) or [`UCS`](ABENUCS2_GLOSRY.html)\\ [character representation](ABENCHAR_REPRESENTATION_GLOSRY.html).

**Unicode check**

[Syntax check](ABENSYNTAX_CHECK_GLOSRY.html) for [ABAP programs](ABENABAP_PROGRAM_GLOSRY.html) that are to be executed in a [Unicode system](ABENUNICODE_SYSTEM_GLOSRY.html). The Unicode check is carried out for [ABAP language versions](ABENABAP_VERSION_GLOSRY.html)\\ [Standard ABAP](ABENSTANDARD_ABAP_GLOSRY.html) or higher. Only Unicode systems are supported in the current [AS ABAP](ABENAS_ABAP_GLOSRY.html) release and therefore no programs of the obsolete ABAP language version Obsolete ABAP (Non-Unicode) are supported.

**Unicode program**

[ABAP program](ABENABAP_PROGRAM_GLOSRY.html) written in [ABAP language version](ABENABAP_VERSION_GLOSRY.html)\\ [Standard ABAP](ABENSTANDARD_ABAP_GLOSRY.html) or higher.

**unit test**

A unit test is created and executed by developers to verify the functional correctness of individual software units. A software unit is defined in this case, as a non-trivial code unit in which certain input has a verifiable effect. [Procedures](ABENPROCEDURE_GLOSRY.html) are a typical example for these units in ABAP. Procedures are the smallest functional components of larger units such as [classes](ABENCLASS_GLOSRY.html) or [function pools](ABENFUNCTION_POOL_GLOSRY.html). Unit tests determine during the development and quality assurance stages that the individual components of a larger software unit work correctly. Unit tests are developed and executed in [ABAP Unit](ABENABAP_UNIT_GLOSRY.html).

**unnamed data object**

[Data object](ABENDATA_OBJECT_GLOSRY.html) that cannot be addressed by a name. Unnamed data objects are [literals](ABENABAP_LITERAL_GLOSRY.html) and [anonymous data objects](ABENANONYMOUS_DATA_OBJECT_GLOSRY.html).

**unrestricted ABAP language version**

[ABAP language version](ABENABAP_VERSION_GLOSRY.html) that is not restricted regarding the ABAP language scope and access to other repository objects. [Standard ABAP](ABENSTANDARD_ABAP_GLOSRY.html) is unrestricted. See also [restricted ABAP language versions](ABENRESTRICTED_VERSION_GLOSRY.html).

**untyped literal**

A [literal](ABENLITERAL_GLOSRY.html) whose [data type](ABENDATA_TYPE_GLOSRY.html) is implicitly derived from its value or its delimiter. Both untyped [numeric literals](ABENNUMERIC_LITERAL_GLOSRY.html) and untyped [character literals](ABENCHARACTER_LITERAL_GLOSRY.html) exist. See also [typed literal](ABENTYPED_LITERAL_GLOSRY.html).

**upcast**

Also referred to as a widening cast. [Assignment](ABENASSIGNMENT_GLOSRY.html) between [reference variables](ABENREFERENCE_VARIABLE_GLOSRY.html), where the [static type](ABENSTATIC_TYPE_GLOSRY.html) of the target variables is more general or identical to the static type of the source variables. An upcast does not have to be performed explicitly. See [downcast](ABENDOWN_CAST_GLOSRY.html).

**update**

Execution of update function modules that are registered using `CALL FUNCTION ... IN UPDATE TASK` and triggered by [`COMMIT WORK`](ABAPCOMMIT.html). Updates are used to bundle the changing database accesses of an [SAP LUW](ABENSAP_LUW_GLOSRY.html) in a [database LUW](ABENDATABASE_LUW_GLOSRY.html). See also synchronous, asynchronous, and local updates.

**update function module**

[Function module](ABENFUNCTION_MODULE_GLOSRY.html) for which the property *update module* is marked in the Function Builder. It is the basis of the update function. An update function module generally contains changing database accesses and the statement `CALL FUNCTION ... IN UPDATE TASK` can be used to register it for later execution with the statement [`COMMIT WORK`](ABAPCOMMIT.html). When an update function module is created, it is assigned either a high or low priority level.

**user date**

[Local date](ABENLOCAL_DATE_GLOSRY.html) of an [ABAP user](ABENABAP_USER_GLOSRY.html) in the [user time zone](ABENUSER_TIME_ZONE_GLOSRY.html).

**user name**

Name of an [ABAP user](ABENABAP_USER_GLOSRY.html) that is required for logging on to a [user session](ABENUSER_SESSION_GLOSRY.html) in the [AS ABAP](ABENAS_ABAP_GLOSRY.html). Together with the [password](ABENPASSWORD_GLOSRY.html), the user name allows authorized use of an AS ABAP.

**UUID**

Short for Universal Unique Identifier. Universally unique identifier that is generated in accordance with [OSF standards](http://www.opengroup.org/onlinepubs/9629399/apdxa.htm) from the system time stamp and the network address. In ABAP, a UUID can be created using the [system class](ABENSYSTEM_CLASS_GLOSRY.html)\\ `CL_SYSTEM_UUID` and the [SQL function](ABENSQL_FUNCTION_GLOSRY.html)\\ [`UUID`](ABENSQL_UUID.html).

**value operator**

[Constructor operator](ABENCONSTRUCTOR_OPERATOR_GLOSRY.html)\\ [`VALUE`](ABENCONSTRUCTOR_EXPRESSION_VALUE.html) used to construct values of a [data type](ABENDATA_TYPE_GLOSRY.html).

**value range**

Set of valid values for a [data object](ABENDATA_OBJECT_GLOSRY.html). The values that are valid for a data object are determined by its [data type](ABENDATA_TYPE_GLOSRY.html). In [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html), the term value range indicates a [semantic property](ABENDDIC_DOMAINS_SEMA.html) of a [DDIC domain](ABENDOMAIN_GLOSRY.html).

**value semantics**

Value semantics means that the referenced data object or the dereferenced [reference](ABENREFERENCE_GLOSRY.html) is used when accessing a [deep](ABENDEEP_GLOSRY.html)\\ [data object](ABENDATA_OBJECT_GLOSRY.html). In ABAP, value semantics apply when accessing [dynamic data objects](ABENDYNAMIC_DATA_OBJECT_GLOSRY.html) and [field symbols](ABENFIELD_SYMBOL_GLOSRY.html). See also [reference semantics](ABENREFERENCE_SEMANTICS_GLOSRY.html).

**variable**

[Named data object](ABENNAMED_DATA_OBJECT_GLOSRY.html) whose value can be changed during the runtime of an [ABAP program](ABENABAP_PROGRAM_GLOSRY.html). See also [immutable variable](ABENIMMUTABLE_VARIABLE_GLOSRY.html).

**visibility section**

The [declaration part](ABENDECLARATION_PART_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) can be divided into up to three visibility sections. These areas are [public](ABENPUBLIC_GLOSRY.html), [protected](ABENPROTECTED_GLOSRY.html), or [private](ABENPRIVATE_GLOSRY.html). They define the visibility of the [components](ABENCLASS_COMPONENT_GLOSRY.html) of the class and therefore the [interfaces](ABENINTF_GEN_GLOSRY.html) of the class towards all users allowed by the [package check](ABENPACKAGE_CONCEPT_GLOSRY.html) concept.

**warning**

[Message](ABENMESSAGE_GLOSRY.html) of [message type](ABENMESSAGE_TYPE_GLOSRY.html)\\ *W*. Warnings during PAI processing of dynpros make screen fields ready for input again.

**weak reference**

Reference to an [object](ABENOBJECT_GLOSRY.html) of a [class](ABENCLASS_GLOSRY.html) that is ignored when the [Garbage Collector](ABENGARBAGE_COLLECTOR_GLOSRY.html) is executed, unlike [object references](ABENOBJECT_REFERENCE_GLOSRY.html) in [reference variables](ABENREFERENCE_VARIABLE_GLOSRY.html). An object that is only pointed to by weak references is deleted by the Garbage Collector.

**WebSocket protocol**

`WSP` for short. A TCP based protocol for the bidirectional communication between a client and a server. Using ABAP Push Channels, an [AS ABAP](ABENAS_ABAP_GLOSRY.html) can be used as a server and as a client for the WebSocket protocol.

**wildcard character**

Character that substitutes other content. **work area**

**writable expression**

Expression representing an operand that can be written to. Writable expressions are certain [constructor expressions](ABENCONSTRUCTOR_EXPRESSION_GLOSRY.html) using [`NEW`](ABENNEW_CONSTRUCTOR_PARAMS_CLASS.html) and [`CAST`](ABENCONSTRUCTOR_EXPRESSION_CAST.html) or [table expressions](ABENTABLE_EXPRESSION_GLOSRY.html). A writable expression can be specified in a [result position](ABENRESULT_POSITION_GLOSRY.html).

**write lock**

Area lock on an area instance version in the shared objects memory. A write lock enables to create a new area instance version.

**write position**

[Operand position](ABENOPERAND_POSITION_GLOSRY.html) in which the [operand](ABENABAP_OPERAND_GLOSRY.html) is written to. There are [read/write positions](ABENREAD_WRITE_POSITION_GLOSRY.html), where an operand is read-accessed before writing and [write-only positions](ABENWRITE_ONLY_POSITION_GLOSRY.html) where no read access occurs.

**write-only position**

[Write position](ABENWRITE_POSITION_GLOSRY.html) in which the [operand](ABENABAP_OPERAND_GLOSRY.html) is only written to but not read-accessed. See [declaration position](ABENDECLARATION_POSITION_GLOSRY.html).

The direction of a writer stream always goes toward the data sink. In a writer stream data can only be written.
