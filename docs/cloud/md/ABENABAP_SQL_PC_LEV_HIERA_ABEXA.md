This example compares [parent-child-based hierarchies](ABENPC_BASED_HIERA_GLOSRY.html) with [level-based hierarchies](ABENLEVEL_BASED_HIERA_GLOSRY.html).

The two variants of the hierarchy generator [`HIERARCHY`](ABENSELECT_HIERARCHY_GENERATOR.html) are used with DDIC database tables as data sources to generate a [parent-child-based hierarchy](ABENPC_BASED_HIERA_GLOSRY.html) and a [level-based hierarchy](ABENLEVEL_BASED_HIERA_GLOSRY.html). Both database tables represent similar hierarchical data that are organized differently:

For the [parent-child-based hierarchy](ABENPC_BASED_HIERA_GLOSRY.html), the hierarchy generator `HIERARCHY` is used with [`CHILD TO PARENT ASSOCIATION`](ABENSELECT_HIERARCHY_GEN_ASSOC.html). The CDS view `DEMO_SIMPLE_TREE` is used as the data source. It accesses `DEMO_SIMPLE_TREE` and exposes a hierarchy association for the parent-child relationship between `ID` and `PARENT_ID`.

For the [level-based hierarchy](ABENLEVEL_BASED_HIERA_GLOSRY.html), the hierarchy generator `HIERARCHY` is used with [`LEVELS`](ABENSELECT_HIERARCHY_GEN_LEVEL.html). The database table `DEMO_LEVELD_TREE` is used directly as data source. Its first four columns are specified as level columns.

The results in internal tables `parent_child_based_hiera` and `level_based_hiera` show similarities as well as differences between the generated SQL hierarchies.

Finally, another parent-child-based hierarchy is generated in a [`WITH`](ABAPWITH.html) statement from the internal table `level_based_hiera`. In that result of a level-based hierarchy, The hierarchy columns `HIERARCHY_RANK` and `HIERARCHY_PARENT_RANK` can be used in a parent-child relationship. The respective hierarchy association is exposed by CTE `+cte_source` from which in CTE `+cte_hiera` a parent-child-based hierarchy is generated. The final `SELECT` statement reads the result. In that parent-child-based hierarchy, all hierarchy columns have the same content as in `parent_child_based_hiera`. Only the content of the data column is different.

-   The three columns of `DEMO_SIMPLE_TREE` are filled with data where the columns `ID` and `PARENT_ID` are suitable for a parent-child relationship for a tree hierarchy and column `NAME` represents arbitrary data.
-   The four columns of `DEMO_LEVELD_TREE` are filled with data where the first four columns are suitable as paths of a tree hierarchy consisting of four levels and column `NAME` represents arbitrary data.

-   The hierarchy columns `HIERARCHY_RANK`, `HIERARCHY_PARENT_RANK`, `HIERARCHY_TREE_SIZE` and `HIERARCHY_LEVEL` are filled in exactly the same way, showing that both results represent similar hierarchies.
-   The hierarchy columns `NODE_ID` and `PARENT_ID` are filled differently.

-   In the parent-child-based hierarchy, they represent the parent-child relationship used for generating the hierarchy.
-   In the level-based hierarchy, they represent the content of the level columns that represent the parent-child relationships for the current an its preceding node.

-   The data column `NAME` has the same content only in the nodes of the highest level 4. For the lower levels:

-   In the parent-child-based hierarchy, the data content is read from the database table.
-   In the level-based hierarchy, the respective rows are not available in the database table but added to the SQL hierarchy. The data column of these rows are set to the null value.

-   The level-based hierarchy has an additional hierarchy column `HIERARCHY_LEVEL_NAME` with the name of the level column that defines the current level.
