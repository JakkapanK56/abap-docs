**Behavior implementation**

For the above RAP behavior definition, one [ABP](ABENBEHAVIOR_POOL_GLOSRY.html) is created. The global class of the behavior pool is `BP_DEMO_MANAGED_ROOT_4`. The actual behavior implementation takes place in local classes that are defined and implemented in the `BP_DEMO_MANAGED_ROOT_4========CCIMP` of the behavior pool. In this case, the behavior pool does not include any implementations that are relevant for the example.

**Access with ABAP using EML**

There are two ABAP EML modify requests that cover several RAP modify operations.

The output shows the database table entries of the root and child entity.

1.  The first ABAP EML modify request covers a RAP create and create-by-association operation. The RAP create operation is based on the construction of an internal table using the [`VALUE`](ABENCONSTRUCTOR_EXPRESSION_VALUE.html) operator. In this case, `%control` is specified and enabled for all fields for all [RAP BO instances](ABENRAP_BO_INSTANCE_GLOSRY.html) after the first bracket. See the concept [here](ABENVALUE_CONSTRUCTOR_PARAMS_LSPC.html). The create-by-association operation does not need to have `%control` specified. The addition [`FIELDS ( ... ) WITH`](ABAPMODIFY_ENTITY_ENTITIES_FIELDS.html) is used which automatically enables `%control`. With a [`COMMIT ENTITIES`](ABAPCOMMIT_ENTITIES.html) statement, the instances are saved to the database.
2.  The second ABAP EML modify request covers a RAP create and update operation. In the create operation, `%control` is specified for each field of the RAP BO instances individually while purposely disabling `%control` for some of the fields. The same is true for the update operation. Here, `%control` is specified using the `VALUE` operator. With a [`COMMIT ENTITIES`](ABAPCOMMIT_ENTITIES.html), the instances are saved to the database.
