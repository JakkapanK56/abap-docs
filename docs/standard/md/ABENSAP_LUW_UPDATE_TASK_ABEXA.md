This example demonstrates how [SAP LUWs](ABENSAP_LUW.html) are bundled using [update function modules](ABENUPDATE_FUNCTION_MODULE_GLOSRY.html).

The example is divided into segments by switching the work process. An implicit [database commit](ABENDATABASE_COMMIT_GLOSRY.html) is performed at the end of each segment. Here, the work process is switched using the statement [`WAIT UP TO`](ABAPWAIT_UP_TO.html). In real programs, a switch of this kind can have [many different causes](ABENDB_COMMIT.html).

This class makes writes on the database that are moved to different [update function modules](ABENUPDATE_FUNCTION_MODULE_GLOSRY.html). The statement [`COMMIT WORK`](ABAPCOMMIT.html) executes the function modules registered so far using the [`CALL FUNCTION IN UPDATE TASK`](ABAPCALL_FUNCTION_UPDATE.html). This entails closing an SAP LUW and opening another.

In the first statement, `COMMIT WORK`, the addition `AND WAIT` triggers [synchronous updates](ABENSYNCHRONOUS_UPDATE_GLOSRY.html), where one update function module must be completed before the next one can start. In the second SAP LUW, the statement [`SET UPDATE TASK`](ABAPSET_UPDATE_TASK_SHORTREF.html) switches [local updates](ABENLOCAL_UPDATE_GLOSRY.html) on, which are always performed synchronously.
