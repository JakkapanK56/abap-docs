[Short Reference](ABAPSELECT_SHORTREF.html)

``... FOR ALL ENTRIES IN @itab WHERE ... col [`operator`](ABENWHERE_LOGEXP_COMPARE.html) @itab-comp ...``

If the addition `FOR ALL ENTRIES` is specified in front of the language element [`WHERE`](ABAPWHERE.html) of the statement `SELECT` of a [main query](ABENMAINQUERY_GLOSRY.html), the components `comp` of the internal table [`itab`](ABENABAP_SQL_HOST_VARIABLES.html) specified here as a [host variable](ABENABAP_SQL_HOST_VARIABLES.html) can be used in [relational expressions](ABENABAP_SQL_STMT_LOGEXP.html) within `[sql_cond](ABENABAP_SQL_STMT_LOGEXP.html)` on the right side of a [comparison operator](ABENWHERE_LOGEXP_COMPARE.html) in comparisons with a [column](ABENABAP_SQL_COLUMNS.html)\\ `col`. The specified component `comp` must be compatible with the column `col`. The internal table `itab` can have a structured or an elementary row type. For an elementary row type, the [pseudo component](ABENPSEUDO_COMPONENT_GLOSRY.html)\\ `table_line` must be specified for `comp`.

The entire logical expression [`sql_cond`](ABENABAP_SQL_STMT_LOGEXP.html) is evaluated for each individual row of the internal table `itab`. The result set of the `SELECT` statement is the union set of the result sets produced by the individual evaluations. Rows that occur more than once are removed from the result set automatically. The entire content of a row is considered here.

**Caution** If the internal table *itab* is empty, the entire *WHERE* condition is ignored. This means that none of the rows in the database table are skipped and are placed in the result set once any duplicate rows are removed.

The logical expression [`sql_cond`](ABENASQL_COND.html) of the `WHERE` condition can comprise multiple [relational expressions](ABENABAP_SQL_STMT_LOGEXP.html) using `AND` and `OR`. However, if `FOR ALL ENTRIES` is specified, there must be at least one [comparison](ABENWHERE_LOGEXP_COMPARE.html) with a column of the internal table `itab` that can be specified statically or dynamically.

The following restrictions apply when using the addition `FOR ALL ENTRIES` with other additions:

The internal table `itab` is evaluated once for each query. Any changes made to the content of the internal table in a `SELECT` loop or `WITH` loop are ignored by the logical expression.

Gets all flight data for a specified departure city. The relevant airlines and flight numbers are first passed to an internal table `entry_tab`, which is evaluated in the `WHERE` condition of the subsequent `SELECT` statement. This selection could also be carried out in a single `SELECT` statement by using a join in the [`FROM`Â clause](ABAPFROM_CLAUSE.html). The table `entry_tab` must not be initial before the `SELECT` statement is executed using `FOR ALL ENTRIES`. The addition [`DISTINCT`](ABAPSELECT_CLAUSE.html) is used to explicitly express the implicit behavior.

Use of `FOR ALL ENTRIES` with an empty internal table. All rows of the DDIC database table are read. The number of read rows is usually smaller in the second `SELECT` statement than in the first statement. This is because only one column is read, and hence more duplicate rows can be removed. The first `SELECT` statement, on the other hand, reads all rows of the DDIC database table to the result set, since the columns cover the entire table key.

-   The addition `FOR ALL ENTRIES` is only possible in front of `WHERE` conditions in a standalone [`SELECT`](ABAPSELECT.html) statement or in the [main query](ABENMAINQUERY_GLOSRY.html) after [`OPEN CURSOR`](ABAPOPEN_CURSOR.html), if no common table expressions are defined using [`WITH`](ABAPWITH.html).
-   The addition `FOR ALL ENTRIES` cannot be used with the addition [`SINGLE`](ABAPSELECT_SINGLE.html).
-   The addition `FOR ALL ENTRIES` cannot be used in combination with [SQL expressions](ABAPSQL_EXPR.html), except for columns specified individually or an aggregate expression `COUNT( * )` specified individually.
-   If the addition `FOR ALL ENTRIES` is used, no [LOB handles](ABENSELECT_INTO_LOB_HANDLES.html) can be created as [reader streams](ABENREADER_STREAM_GLOSRY.html) or as [locators](ABENLOCATOR_GLOSRY.html) in the [target area](ABAPINTO_CLAUSE.html) of a standalone [`SELECT`](ABAPSELECT.html) statement.
-   The addition `FOR ALL ENTRIES` cannot be combined with [`UNION`](ABAPUNION.html), [`INTERSECT`](ABAPUNION.html), and [`EXCEPT`](ABAPUNION.html).
-   In a `SELECT` statement with `FOR ALL ENTRIES`, no aggregate expressions except for `COUNT( * )` can be used in the [`SELECT`Â list](ABAPSELECT_LIST.html). In cases like these, the aggregate expression is not evaluated in the database, but is emulated on the AS ABAP.
-   The addition `FOR ALL ENTRIES` should not be used with the addition [`GROUP BY`](ABAPGROUPBY_CLAUSE.html). The addition `GROUP BY` has no effect if `FOR ALL ENTRIES` is used.
-   In a `SELECT` statement with `FOR ALL ENTRIES`, the addition [`ORDER BY`](ABAPORDERBY_CLAUSE.html) can only be used with the addition `PRIMARY KEY` and can only be used to access a single table or view. In this case, all columns of the primary key must be in the [`SELECT`Â list](ABAPSELECT_LIST.html), except for the client column in client-dependent tables.
-   No [path expressions](ABENABAP_SQL_PATH.html) can be used in a `SELECT` statement with `FOR ALL ENTRIES`.
-   If the addition `FOR ALL ENTRIES` is used, the [`SELECT`Â list](ABAPSELECT_LIST.html) should not contain any database fields of the built-in types `STRING`, `RAWSTRING`, and `GEOM_EWKB` as well as `LCHR` and `LRAW`. These data types prevent rows that occur more than once on the database system from being removed. These rows are only removed from the result set on the AS ABAP. If listed in the `SELECT` list, a syntax check warning is raised that can be hidden by a pragma.

-   The same internal table can be specified after `FOR ALL ENTRIES` and after [`INTO`](ABAPINTO_CLAUSE.html). The content of the table is evaluated by `FOR ALL ENTRIES` and then overwritten by the [`INTO`Â clause](ABAPINTO_CLAUSE.html).
-   A comparison with a column of an internal table can also be performed using the `WHERE` condition of a [subquery](ABENSUBQUERY_GLOSRY.html) for the same [data source](ABAPSELECT_DATA_SOURCE.html).
-   With respect to rows occurring more than once in the result set, the addition `FOR ALL ENTRIES` has the same effect as when the addition [`DISTINCT`](ABAPSELECT_CLAUSE.html) is specified in the definition of the selection set. Unlike `DISTINCT`, the rows are not always deleted by the database system and are sometimes only deleted from the result set on AS ABAP instead. The duplicate rows are then removed from the database system if the `SELECT` statement can be passed to the database system as a single SQL statement and the addition `DISTINCT` is supported. If the `SELECT` statement needs to be distributed to multiple SQL statements before it is passed or if columns of the types `STRING` and `RAWSTRING` plus `LCHR` and `LRAW` are specified in the [`SELECT`Â list](ABAPSELECT_LIST.html), the rows are aggregated on AS ABAP.
-   If duplicate rows are only removed from AS ABAP, all rows specified by the `WHERE` condition (in some cases) are passed to an internal system table and then aggregated. The [maximum size](ABENMEMORY_CONSUMPTION_2.html) of this system table is restricted to that of regular internal tables. More specifically, the system table is always required if one of the additions [`PACKAGE SIZE`](ABAPINTO_CLAUSE.html), [`UP TO`](ABAPSELECT_UP_TO_OFFSET.html), or [`OFFSET`](ABAPSELECT_UP_TO_OFFSET.html) is used simultaneously. These then have no effect on the number of rows passed from the database server to AS ABAP but are only used when the rows are passed from the system table to the actual target area. If the maximum size of the internal system table is exceeded, a runtime error occurs.
-   If the column types allow it, the addition [`DISTINCT`](ABAPSELECT_CLAUSE.html) might be used together with `FOR ALL ENTRIES` in order to explicitly express the behavior.
-   The addition `FOR ALL ENTRIES` can be processed by the [ABAP SQL in-memory engine](ABENABAP_SQL_INMEMENG.html) for
-   accessing the [table buffer](ABENTABLE_BUFFER_GLOSRY.html) and an internal table with [`FROM @itab`](ABAPSELECT_ITAB.html). The [table buffer](ABENTABLE_BUFFER_GLOSRY.html) is bypassed only for tables with generic buffering if the condition after `FOR ALL ENTRIES` prevents a single generic area from being specified exactly.
-   For buffered tables, the addition `FOR ALL ENTRIES` can be a more efficient alternative to [join expressions](ABENJOIN_EXPRESSION_GLOSRY.html).
-   It is strongly recommended that the internal table `itab` is not initial before using it after `FOR ALL ENTRIES`. In case of an initial internal table, all rows are read from the database regardless of any further conditions specified after `WHERE`, which is generally not desired.
-   It depends on the database platform and different settings, how the content of the internal table is passed to the database and where it is transformed into conditions. If [FDA](ABENFAST_DATA_ACCESS_GLOSRY.html) write access is available, it can be used by `FOR ALL ENTRIES`.
-   If, in a [strict mode of the syntax check](ABENABAP_SQL_STRICT_MODES.html), `FOR ALL ENTRIES` is specified together with columns of the types `STRING` and `RAWSTRING` plus `LCHR` and `LRAW` in the [`SELECT`Â list](ABAPSELECT_LIST.html), the syntax check is performed in [strict mode from ABAP release 7.68](ABENABAP_SQL_STRICTMODE_768.html) .
-   If the entire `WHERE` condition is ignored because the internal table `itab` is empty, the implicit `WHERE` condition for the [current client](ABENCURRENT_CLIENT_GLOSRY.html) or the client specified using [`USING CLIENT`](ABAPSELECT_CLIENT.html) is not affected, if [implicit client handling](ABENABAP_SQL_CLIENT_HANDLING.html) is enabled. This means that all data is only read from the current client.
-   If [implicit client handling](ABENABAP_SQL_CLIENT_HANDLING.html) is disabled using the obsolete addition [`CLIENT SPECIFIED`](ABAPSELECT_CLIENT_OBSOLETE.html), no implicit `WHERE` condition exists for the client. Any `WHERE` condition specified explicitly for the client column is ignored with the entire condition if the internal table `itab` is empty and the data from all clients is read.
