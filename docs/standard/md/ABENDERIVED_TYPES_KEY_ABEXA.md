This example demonstrates the use of `%key` with a [managed](ABENBDL_IMPL_TYPE.html)\\ [RAP BO](ABENRAP_BO_GLOSRY.html) in the [RAP interaction phase](ABENRAP_INT_PHASE_GLOSRY.html).

**Data model**

The CDS data model consists of the root entity `DEMO_MANAGED_ROOT_4` and its child entity `DEMO_MANAGED_CHILD_4`. The child entity is not used in the example.

The [RAP behavior definition](ABENCDS_BEHAVIOR_DEFINITION_GLOSRY.html)\\ `DEMO_MANAGED_ROOT_4` is defined in [RAP BDL](ABENCDS_BDL_GLOSRY.html) as follows:

**Behavior implementation**

For the above RAP behavior definition, one [ABP](ABENBEHAVIOR_POOL_GLOSRY.html) is created. The global class of the behavior pool is `BP_DEMO_MANAGED_ROOT_4`. The actual behavior implementation takes place in local classes that are defined and implemented in the `BP_DEMO_MANAGED_ROOT_4========CCIMP` of the behavior pool. In this case, the behavior pool does not include any implementations that are relevant for the example.

**Access with ABAP using EML**

There is one ABAP EML modify request that covers a RAP create and update operation. In both create and update operation, `%key` is specified in different ways just to demonstrate the variety of specification options of this [component group](ABENCOMPONENT_GROUP_GLOSRY.html). See more information on this in the topic [Using Keys and Identifying RAP BO Instances in a Nutshell](ABAPDERIVED_TYPES_KEYS_IDENTIFIERS.html).

With a [`COMMIT ENTITIES`](ABAPCOMMIT_ENTITIES.html) statement, the instances are saved to the database. The output shows the database table entries as well as the mapped and failed responses for the root entity.
