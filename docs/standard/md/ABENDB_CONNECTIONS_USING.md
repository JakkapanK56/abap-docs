Secondary connections and service connections are opened and used in different ways in ABAP SQL, Native SQL, and AMDP. A secondary or service connection active for an internal session can be shared by ABAP SQL, Native SQL, or AMDP. Connections used by ABAP SQL, however, cannot have any lowercase letters in their names and ADBC can reserve connections for itself exclusively. Service connections represented by [ABAP SQL connection object](ABENABAP_SQL_CONN_OBJ_GLOSRY.html) are never shared.

The following additions can be used to use a database connection other than the [standard connection](ABENSTANDARD_DB_CONNECTION_GLOSRY.html) in ABAP SQL:

Here, a secondary connection or a service connection can be specified by using an [ABAP SQL connection object](ABENABAP_SQL_CONN_OBJ_GLOSRY.html) or its name.

If the addition `CONNECTION` is specified using the name of a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) from the database table `DBCON` or using the name of a [service connection](ABENSERVICE_CONNECTION_GLOSRY.html) in an ABAP SQL statement with `con|(con_syntax)`, the following applies:

If the addition `CONNECTION` is specified using a reference variable `con_ref` to an [ABAP SQL connection object](ABENABAP_SQL_CONN_OBJ_GLOSRY.html), the connection is handled by the connection object only.

When a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) is used, an ABAP SQL statement only accesses the [database schema](ABENDATABASE_SCHEMA_GLOSRY.html) assigned to the [database user name](ABENDATABASE_USER_NAME_GLOSRY.html) assigned to the secondary connection. ABAP SQL can only access database tables and views that are defined in the [ABAP Dictionary](ABENABAP_DICTIONARY_GLOSRY.html) in the current AS ABAP, which means that every database table or view accessed using a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) must exist and be active as a database or view with the same name and matching DDIC structure in the current AS ABAP, regardless of the [secondary database](ABENSECONDARY_DB_GLOSRY.html) and database schema. For the database tables or views of the secondary database, ABAP SQL assumes that their [table category](ABENDDIC_DATABASE_TABLES_TAB_CAT.html) and type information matches the local definition exactly. This prerequisite is essential, for example, for the correct interpretation of the database content and, if necessary, their conversion with respect to the ABAP types of the target fields. If this prerequisite is not met, incorrect data or runtime errors can be caused by reads or writes depending on the database system. Because the ABAP runtime framework cannot ensure the consistency of the type descriptions in the local and remote databases, the consistency must be guaranteed by the relevant application program.

The name of a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) or [service connection](ABENSERVICE_CONNECTION_GLOSRY.html) specified after `CONNECTION` is transformed into uppercase letters internally. This must be respected when Native SQL accesses the connection explicitly. Conversely, an ABAP SQL statement can reuse database connections active in Native SQL or AMDP only if their names do not contain any lowercase letters.

Specification of a database connection in ABAP SQL by using its name.

See [ABAP SQL - Connection Objects](ABENIF_ABAP_SQL_CONNECTION_ABEXA.html)

To use a database connection other than the [standard connection](ABENSTANDARD_DB_CONNECTION_GLOSRY.html) in [ADBC](ABENADBC_GLOSRY.html), the static methods [`ABAP`](ABENIF_SQL_CONNECTION.html) or [`PRIVATE`](ABENIF_SQL_CONNECTION.html) of the class `CL_SQL_ADBC_CONNECTION_BUILDER` can be used. The return value of the method is a reference to a connection object that can be passed to other ADBC objects. Using the input parameter `I_NAME`, the method can be passed the name of a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) from the database table `DBCON` or a [service connection](ABENSERVICE_CONNECTION_GLOSRY.html). These names are case-sensitive. The activated connection is an [ABAP connection](ABENABAP_CONNECTION_GLOSRY.html) that can be used in ABAP SQL, Native SQL, and AMDP.

After successful execution, the connection gets a reference to a connection object. The connection object represents the connection activated or reused in ADBC and can be passed to a regular or prepared statement. The SQL statements of these classes are then executed on this connection in its database LUW.

Specification of a database connection in ADBC.

The following special statements can be used to use a database connection other than the [standard connection](ABENSTANDARD_DB_CONNECTION_GLOSRY.html) in Native SQL embedded between [`EXEC SQL`](ABAPEXEC.html) and [`ENDEXEC`](ABAPENDEXEC.html)

The statement [`CONNECT TO`](ABAPEXEC_CONNECTION.html) can be used to activate a [secondary connection](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) or a [service connection](ABENSERVICE_CONNECTION_GLOSRY.html). Here, the addition `AS` can be used to open the same connection more than once under different names. The entries made here are case-sensitive.

After the statement `CONNECT TO`, all subsequent Native SQL statements embedded between [`EXEC SQL`](ABAPEXEC.html) and [`ENDEXEC`](ABAPENDEXEC.html) in the current internal session use the connection activated or reused using `CONNECT TO`. The statement [`SET CONNECTION`](ABAPEXEC_CONNECTION.html) can be used to switch to the use of the previously activated connection of the current internal session. Here, a name defined with `AS` must be specified using this connection. More specifically, the [standard connection](ABENSTANDARD_DB_CONNECTION_GLOSRY.html) can be activated by specifying `DEFAULT`. The statement `SET CONNECTION` is ignored by the database LUWs of the connections involved.

When a name is specified after `AS` in the statement `CONNECT TO`, multiple database connections with the same original name can be activated and used in the same internal session. These are indicated in the output of the program `DBCONINFO` by the different values in the column `Hdl`.

Specification of a database connection after `EXEC SQL`.

To use a database connection other than the [standard connection](ABENSTANDARD_DB_CONNECTION_GLOSRY.html) to execute an [AMDP procedure implementation](ABENAMDP_PROCEDURE_METHOD_GLOSRY.html), the names of [service connections](ABENSERVICE_CONNECTION_GLOSRY.html) can be passed to the input parameter [`CONNECTION`](ABENAMDP_DB_CONNECTIONS.html) of the associated [AMDP method](ABENAMDP_METHOD_GLOSRY.html). The names are case-sensitive here. [Secondary connections](ABENSECONDARY_DB_CONNECTION_GLOSRY.html) cannot be used.

The service connections that can be used by AMDP are also connections that can be used by ABAP SQL, Native SQL, and AMDP, as long as the appropriate conditions are met.
